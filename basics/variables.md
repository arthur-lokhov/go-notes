# 🔧 Переменные в Go

Представь, что ты переезжаешь в новую квартиру. У тебя есть коробки с вещами. Некоторые коробки ты подписываешь — «кухня», «спальня», «книги». Это помогает найти нужную вещь, не перебирая всё подряд.

**Переменная в Go** — это такая же подписанная коробка. Имя переменной — это надпись на коробке. Тип — что внутри можно хранить. Значение — то, что сейчас лежит внутри.

Но в Go есть нюанс: ты не всегда знаешь, где будет стоять коробка — на полке в прихожей (стек) или на складе в подвале (куча). И это решает не ты, а компилятор.

---

## 🧠 Как Go хранит переменные

Когда ты объявляешь переменную, Go резервирует область памяти. Размер этой области зависит от типа: `int` займёт 8 байт на 64-битной системе, `bool` — 1 байт, структура — сколько получится с учётом выравнивания.

**Но где именно?**

Здесь начинается самое интересное. В Go есть два места для хранения:

**Стек** — это быстрая, упорядоченная память. Представь стопку тарелок: кладёшь сверху, берёшь сверху. Когда функция завершается, всё, что было в её стеке, автоматически очищается. Быстро, эффективно, без накладных расходов.

**Куча** — это большая, но медленная память. Здесь живут объекты, которые должны пережить функцию. Но за это приходится платить: кучу убирает сборщик мусора, а это дополнительные циклы процессора.

### Распространённое заблуждение

Многие думают: «значимые типы (`int`, `struct`) живут на стеке, ссылочные (`slice`, `map`) — в куче».

**Это не так.**

`int` может оказаться в куче. `map` может остаться на стеке. Всё решает **escape analysis** — анализ побега.

---

## 🔬 Escape Analysis: как компилятор решает

**Escape analysis** — это процесс, при котором компилятор определяет: может ли переменная «сбежать» из функции наружу?

Если компилятор может доказать, что переменная не нужна после завершения функции — она остаётся на стеке. Если нет — сбегает в кучу.

**Пример побега:**

```go
func createUser() *User {
    u := User{Name: "Alice"}
    return &u  // u сбегает в кучу!
}
```

Здесь `u` не может остаться на стеке. Когда функция завершится, стек очистится. Но указатель на `u` возвращается наружу — значит, `u` должен жить дальше. Компилятор размещает `u` в куче.

**Пример без побега:**

```go
func calculate() int {
    x := 10
    y := 20
    return x + y  // x и y остаются на стеке
}
```

Здесь `x` и `y` не «убегают» — они используются только внутри функции. Компилятор оставляет их на стеке.

### Как посмотреть решения компилятора

```bash
go build -gcflags="-m" main.go
```

Вывод скажет тебе правду:

```
./main.go:5:9: &u escapes to heap
./main.go:10:5: x does not escape
```

Компилятор буквально говорит: «эта переменная сбежала в кучу, а эта осталась на стеке».

## 📦 Как объявлять переменные

В Go есть несколько способов объявить переменную. Каждый — для своего случая.

### Классическое `var`

```go
var name Type = value
```

Это полная форма. Ты явно указываешь тип и значение. Если не указать значение — переменная получит **нулевое значение** для своего типа: `0` для чисел, `false` для `bool`, `""` для строк, `nil` для указателей.

```go
var count int        // count = 0
var done bool        // done = false
var name string      // name = ""
var ptr *int         // ptr = nil
```

**Когда использовать `var`:**

- Когда нужна переменная с нулевым значением
- Когда тип не очевиден и его лучше указать явно
- Для глобальных переменных (на уровне пакета)

### Краткое объявление `:=`

```go
message := "hello"
```

Это сахар. Компилятор сам выводит тип. Внутри функции это эквивалентно `var message = "hello"`.

**Важное правило:** `:=` работает только внутри функций. На уровне пакета используй `var`.

### Множественное объявление

Go позволяет объявлять несколько переменных сразу:

```go
a, b := 1, "hello"
```

Это особенно удобно, когда функция возвращает несколько значений:

```go
value, err := getValue()
```

## ⚠️ `=` vs `:=` — в чём разница

Это один из самых частых источников ошибок.

**`=`** — присваивание. Ты меняешь значение существующей переменной.

**`:=`** — объявление **и** присваивание. Ты создаёшь новую переменную.

**Подводный камень:** `:=` может создать новую переменную там, где ты не ожидаешь.

```go
x := 1
x, y := 5, 10  // x — НОВАЯ переменная!
fmt.Println(x) // 5, а не 1!
```

Здесь компилятор видит: `y` — новая переменная, значит можно использовать `:=`. Но `x` тоже объявляется заново — происходит **shadowing** (затенение).

**Как избежать:** если переменная уже существует, используй `=`:

```go
x := 1
x = 5      // присваивание
y := 10    // объявление
```

## 🎭 Shadowing: когда переменные прячутся

**Shadowing** — это когда переменная из внутренней области видимости скрывает переменную с таким же именем из внешней.

```go
x := 1
{
    x := 2  // новая переменная, скрывает внешнюю
    fmt.Println(x)  // 2
}
fmt.Println(x)  // 1 — внешняя не изменилась
```

Это не всегда ошибка. Иногда так делают намеренно. Но чаще — это источник багов.

### Классическая ошибка с `err`

Проблема возникает, когда `:=` создаёт новую переменную `err` внутри блока, а потом ты пытаешься проверить ошибку **после** этого блока:

```go
func process() error {
    file, err := os.Open("file.txt")  // outer err
    if err != nil {
        return err
    }
    defer file.Close()

    // ← Внимание: := создаёт НОВУЮ переменную err!
    if _, err := io.WriteString(file, "Hello"); err != nil {
        log.Printf("Ошибка: %v", err)  // ← inner err (видна только здесь)
        // ← Забыли return err!
    }
    
    // ← Сюда код ДОХОДИТ даже при ошибке!
    if err != nil {  // ← ПРОВЕРЯЕМ outer err (всё ещё nil!)
        // ← Это НИКОГДА не выполнится при ошибке WriteString
    }
    
    return err  // ← Возвращаем nil, даже если WriteString failed!
}
```

**Что происходит:**

```
┌─────────────────────────────────────────────────────────────┐
│  func process() error {                                     │
│      ┌─────────────────────────────────────────────────┐    │
│      │  err (внешняя) = nil                            │    │
│      │  file, err := os.Open(...)                      │    │
│      │                                                 │    │
│      │  if _, err := WriteString(...) {                │    │
│      │      ┌─────────────────────────────────────┐    │    │
│      │      │  err (внутренняя) = ошибка или nil  │    │    │
│      │      │  ← СКРЫВАЕТ внешнюю err!            │    │    │
│      │      │  log.Printf(...) ← видит inner err  │    │    │
│      │      └─────────────────────────────────────┘    │    │
│      │  }                                              │    │
│      │                                                 │    │
│      │  if err != nil {  ← outer err (всё ещё nil!)    │    │
│      │      // ← Никогда не выполнится при ошибке!     │    │
│      │  }                                              │    │
│      │                                                 │    │
│      │  return err  ← Возвращает outer err (nil)!      │    │
│      └─────────────────────────────────────────────────┘    │
│  }                                                          │
└─────────────────────────────────────────────────────────────┘
```

**Почему это баг:** если `WriteString` вернул ошибку, мы её залогировали, но **не вернули из функции**. А проверка `if err != nil` после блока проверяет **внешнюю** `err`, которая всё ещё `nil`.

**Как исправить:**

```go
func process() error {
    file, err := os.Open("file.txt")
    if err != nil {
        return err
    }
    defer file.Close()

    // ← Используем = вместо :=
    _, err = io.WriteString(file, "Hello")
    if err != nil {
        log.Printf("Ошибка: %v", err)
        return err  // ← Возвращаем ошибку
    }
    
    return nil  // ← Успех
}
```

Или короче:

```go
func process() error {
    file, err := os.Open("file.txt")
    if err != nil {
        return err
    }
    defer file.Close()

    _, err = io.WriteString(file, "Hello")
    return err  // ← Вернёт ошибку или nil
}
```

**Правило:** если планируешь использовать переменную после блока `if`, не создавай новую переменную внутри `if` через `:=`.

## 🌐 Глобальные переменные

**Глобальные переменные** — объявлены на уровне пакета, вне функций. Видны во всех файлах пакета.

```go
// config.go
package config

var Port = 8080  // глобальная переменная
```

**Проблема:** глобальные переменные усложняют тестирование и понимание кода. Ты не знаешь, где и когда они изменятся.

**Лучшая альтернатива:** явные конструкторы.

```go
// ❌ Плохо
var db *sql.DB

func init() {
    db = connectDB()  // скрытая инициализация
}

// ✅ Лучше
type Config struct {
    DB *sql.DB
}

func NewConfig() *Config {
    return &Config{
        DB: connectDB(),  // явная инициализация
    }
}
```

## 🔟 Нулевые значения

В Go у каждой переменной есть **нулевое значение** по умолчанию. Это не «мусор в памяти», а предсказуемое значение.

| Тип | Нулевое значение |
|-----|------------------|
| `int`, `float` | `0` |
| `bool` | `false` |
| `string` | `""` |
| `pointer`, `slice`, `map`, `chan`, `interface`, `func` | `nil` |
| `struct` | все поля = их нулевые значения |

**Почему это важно:** ты можешь использовать переменную сразу после объявления, не инициализируя.

```go
var count int
count++  // ✅ count = 0, потом 1
```

## 🧬 Внутреннее устройство

### Как компилятор видит переменные

Внутри компилятора Go каждая переменная представлена интерфейсом `types.Object`. Это унифицированное представление для всех именованных сущностей: переменных, функций, констант, типов.

```go
type Object interface {
    Parent() *Scope      // Область видимости
    Pos() token.Pos      // Позиция в исходнике
    Pkg() *Package       // Пакет
    Name() string        // Имя
    Type() Type          // Тип
    Exported() bool      // Экспортируется ли
}
```

**Конкретно переменные** представлены структурой `types.Var`:

```go
type Var struct {
    object
    origin   *Var        // Исходная переменная (для generics)
    kind     VarKind     // Контекст: локальная, параметр, поле
    embedded bool        // Встроенное поле
}
```

**`VarKind`** говорит компилятору, где используется переменная:

```go
type VarKind uint8

const (
    _          VarKind = iota
    PackageVar          // Глобальная переменная
    LocalVar            // Локальная переменная
    RecvVar             // Ресивер метода
    ParamVar            // Параметр функции
    ResultVar           // Возвращаемое значение
    FieldVar            // Поле структуры
)
```

**Зачем это знать?** Если ты пишешь инструменты для Go (линтеры, анализаторы), ты будешь работать с этими структурами. Для обычного кода достаточно понимать: компилятор видит переменные не как «ячейки памяти», а как объекты с метаданными.

## 🏆 Лучшие практики

**1. Объявляй переменные как можно ближе к использованию**

```go
// ❌ Плохо
var count int
// ... 50 строк кода ...
count = 10

// ✅ Лучше
// ... 50 строк кода ...
count := 10
```

**2. Предпочитай `:=` внутри функций**

Краткое объявление делает код лаконичнее. Но помни про shadowing.

**3. Избегай глобальных переменных**

Используй их только для действительно глобальных состояний: логгер, конфигурация, пул соединений.

**4. Минимизируй область видимости**

Паттерн `if v := f(); err == nil { ... }` — хороший пример. Переменная `v` живёт только там, где нужна.

**5. Понимай, когда переменная убегает в кучу**

Если производительность критична, проверяй escape analysis:

```bash
go build -gcflags="-m" main.go
```

Если видишь `escapes to heap` — подумай, можно ли переписать код, чтобы переменная осталась на стеке.

---

## 🎯 Итог

Переменные в Go — это не просто «ячейки памяти». Это объект с метаданными, который компилятор размещает на стеке или в куче в зависимости от того, как ты его используешь.

**Запомни:**

- Escape analysis решает, где жить переменной
- `:=` объявляет **и** присваивает, `=` только присваивает
- Shadowing — частый источник багов
- `nil` ведёт себя по-разному для разных типов
- Глобальные переменные — зло, если не использовать с осторожностью

Понимание того, как работают переменные, сэкономит тебе часы отладки и поможет писать более эффективный код.
