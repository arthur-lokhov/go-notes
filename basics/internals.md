# ⚙️ Internals: как Go собирает и оптимизирует код

Если бы ты был волшебником, то обычный код — это заклинания, которые ты произносишь. А internals — это понимание того, как магия работает внутри: какие ингредиенты смешиваются, какие символы рисуются, откуда берётся сила.

Большинство разработчиков пишут код и нажимают `go build`. Всё работает. Но когда что-то идёт не так — сборка медленная, бинарник огромный, производительность хромает — нужно понимать, что происходит под капотом.

Эта глава — про то, как Go превращает твой код в работающую программу.

## 🏛 GOROOT, GOPATH, GO111MODULE

Прежде чем говорить о сборке, нужно понять, где Go ищет код.

### GOROOT

**GOROOT** — это директория, где установлен сам Go.

```bash
go env GOROOT
# /usr/local/go (на macOS)
# /usr/lib/go (на Linux)
```

В GOROOT лежит:
- Компилятор (`pkg/tool/`)
- Стандартная библиотека (`src/`, `pkg/`)
- Документация

**Обычно не трогай GOROOT.** Он настроен при установке.

### GOPATH

**GOPATH** — это рабочее пространство для Go-проектов (до модулей).

```bash
go env GOPATH
# $HOME/go
```

Структура GOPATH:

```
$GOPATH/
├── src/          # Исходный код
├── pkg/          # Скомпилированные пакеты
└── bin/          # Бинарники
```

**С появлением модулей GOPATH устарел.** Теперь проекты могут быть где угодно. Но GOPATH всё ещё используется для:
- Кэша модулей (`$GOPATH/pkg/mod`)
- Бинарников (`$GOPATH/bin`)

### GO111MODULE

**GO111MODULE** — переменная окружения, которая управляет режимом работы модулей:

| Значение | Поведение |
|----------|-----------|
| `auto` (по умолчанию) | Модули включены, если есть `go.mod` |
| `on` | Модули всегда включены |
| `off` | Модули выключены, используется GOPATH |

**Современная рекомендация:** не устанавливать GO111MODULE, оставить `auto`.

## 📦 Go Modules: как работают зависимости

### go.mod и go.sum

Мы уже говорили о модулях в [предыдущей главе](./packages-modules-workspaces.md). Здесь разберём internals.

**go.mod** описывает модуль:

```gomod
module github.com/myorg/myproject

go 1.22

require (
    github.com/google/uuid v1.3.0
    github.com/stretchr/testify v1.8.0 // indirect
)
```

**go.sum** содержит хэши для проверки целостности:

```
github.com/google/uuid v1.3.0 h1:t6JcX0F+Vn90vG8bVn1fGf9G8rX8H8L8V8H8H8H8H8=
github.com/google/uuid v1.3.0/go.mod h1:TIyPZe4MgqvfeYDBFedMoGGpEw/LqOeaOT+nhxU+yHo=
```

### Как Go загружает зависимости

Когда ты выполняешь `go build`:

```
┌─────────────────────────────────────────────────────────┐
│  1. Проверяет локальный кэш ($GOPATH/pkg/mod)           │
│     ↓ не найдено                                        │
│  2. Запрашивает proxy.golang.org                        │
│     ↓                                                   │
│  3. Загружает модуль                                    │
│     ↓                                                   │
│  4. Сверяет хэш с go.sum (sum.golang.org)               │
│     ↓                                                   │
│  5. Кэширует локально                                   │
│     ↓                                                   │
│  6. Компилирует                                         │
└─────────────────────────────────────────────────────────┘
```

**proxy.golang.org** — прокси-сервер Google, кэширующий модули.

**sum.golang.org** — база данных контрольных сумм для проверки целостности.

## 🔨 Build Process: процесс сборки

### Компиляция по пакетам

Go компилирует код **по пакетам**, а не по файлам. Каждый пакет компилируется отдельно, результат сохраняется в кэш, затем линковщик собирает всё вместе.

```
┌─────────────────────────────────────────────────────────┐
│  go build                                               │
│  ↓                                                      │
│  1. Компиляция пакета util → util.a (кэш)               │
│  ↓                                                      │
│  2. Компиляция пакета config → config.a (кэш)           │
│  ↓                                                      │
│  3. Компиляция пакета main → main.a                     │
│  ↓                                                      │
│  4. Линковка: util.a + config.a + main.a → бинарник     │
└─────────────────────────────────────────────────────────┘
```

**Кэш компиляции** находится в `GOCACHE`:

```bash
go env GOCACHE
# $HOME/Library/Caches/go-build (macOS)
# $HOME/.cache/go-build (Linux)
```

При повторной сборке, если исходники не изменились, Go берёт пакет из кэша.

### Просмотр процесса сборки

```bash
go build -x  # показывает команды компиляции
go build -v  # показывает компилируемые пакеты
go build -a  # пересобирает всё, игнорируя кэш
```

Пример вывода `go build -x`:

```
WORK=/var/folders/.../go-build123456
cd /path/to/project
/usr/local/go/pkg/tool/darwin_amd64/compile -o $WORK/b001/_pkg_.a ...
/usr/local/go/pkg/tool/darwin_amd64/link -o project $WORK/b001/_pkg_.a ...
```

### .a-файлы: объектные архивы

Результат компиляции пакета — это `.a`-файл (archive):

```
$GOCACHE/
└── aa/
    └── aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa-d  # util.a
```

Это не обычный архив, а специальный формат Go, содержащий:
- Скомпилированный код
- Метаданные (типы, имена экспортируемых символов)
- Отладочную информацию

### Как `go build` определяет имя выходного файла

Поведение зависит от того, что ты компилируешь:

**1. Главный пакет (main):**

```bash
go build example.com/myapp
# → myapp (или myapp.exe на Windows)

go build example.com/myapp/v2
# → myapp (не v2!)
```

Имя берётся из последнего компонента import path без версии.

**2. Список .go файлов:**

```bash
go build ed.go rx.go
# → ed (по первому файлу)
```

**3. Не-main пакет:**

```bash
go build example.com/mylib
# → ничего не создаётся, только проверка что компилируется
```

**4. Флаг `-o`:**

```bash
go build -o myapp cmd/main.go
# → принудительно задаёт имя
```

### Режимы сборки: `-buildmode`

Флаг `-buildmode` определяет, какой тип объекта создавать. Это **не флаг линковщика**, а флаг сборки — он влияет на весь процесс компиляции.

| Режим | Что создаёт | Когда использовать |
|-------|-------------|-------------------|
| `default` | .exe для main, .a для остальных | Обычная сборка |
| `archive` | .a файлы | Библиотеки для C |
| `c-archive` | .a + .h заголовок | Интеграция с C |
| `c-shared` | .so/.dll библиотека | Shared library для C |
| `shared` | Общая .so для нескольких Go программ | Экономия памяти |
| `exe` | .exe всегда | Принудительный exe |
| `pie` | PIE executable | Повышение безопасности |
| `plugin` | .so плагин | Динамическая загрузка |

**Примеры:**

```bash
# C-библиотека из Go кода
go build -buildmode=c-archive -o libmylib.a mylib.go
# Создаёт: libmylib.a + libmylib.h

# PIE бинарник для безопасности
go build -buildmode=pie -o myapp

# Плагин для динамической загрузки
go build -buildmode=plugin -o myplugin.so
```

### Полезные флаги `go build`

**Основные:**

```bash
go build -v          # показывать компилируемые пакеты
go build -x          # показывать команды компиляции
go build -a          # пересобрать всё, игнорируя кэш
go build -n          # показать команды, но не выполнять
go build -o myapp    # задать имя выходного файла
```

**Параллелизм:**

```bash
go build -p 4        # сколько сборок запускать параллельно
# По умолчанию: GOMAXPROCS (число CPU)
```

**Отладка и проверка:**

```bash
go build -race       # включить race detector
go build -msan       # memory sanitizer
go build -asan       # address sanitizer
go build -cover      # code coverage
```

**Модули и пути:**

```bash
go build -mod=readonly   # не менять go.mod
go build -mod=vendor     # использовать vendor/
go build -trimpath       # убрать пути из бинарника
go build -buildvcs=false # не добавлять VCS информацию
```

**Передача флагов инструментам:**

```bash
go build -gcflags="-m"           # флаги компилятору
go build -ldflags="-s -w"        # флаги линковщику
go build -asmflags="-S"          # флаги ассемблеру
go build -toolexec='trace -'     # обёртка на инструменты
```

## 🔧 Компилятор (go tool compile): gcflags

**Компилятор Go** (`go tool compile`) превращает исходный код в объектные файлы. Флаг `-gcflags` передаёт параметры компилятору.

### Как посмотреть все флаги компилятора

```bash
go tool compile -help
```

Вывод покажет все доступные опции. Вот только те, что имеют практическую ценность:

| Флаг | Описание | Пример |
|------|----------|--------|
| `-m` | Печатает решения об inlining и escape analysis | `-gcflags="-m"` |
| `-S` | Выводит SSA representation | `-gcflags="-S"` |
| `-l` | Запрещает inlining | `-gcflags="-l"` |
| `-d=...` | Внутренние отладочные флаги | `-gcflags="-d=esc"` |
| `-N` | Отключает оптимизации | `-gcflags="-N"` |
| `-B` | Отключает проверку границ | `-gcflags="-B"` |
| `-race` | Включает race detector | `-gcflags="-race"` |

### Популярные комбинации

```bash
# Показать inlining и escape analysis
go build -gcflags="-m" main.go

# Больше деталей об escape analysis
go build -gcflags="-m -m" main.go

# Показать SSA (в консоль)
go build -gcflags="-S" main.go

# Запретить inlining
go build -gcflags="-l" main.go

# Проверка указателей
go build -gcflags="-d=checkptr" main.go

# Детальный escape analysis
go build -gcflags="-d=esc" main.go

# Отладка softfloat
go build -gcflags="-d=softfloat" main.go

# Проверка GC
go build -gcflags="-d=gccheckmark" main.go

# Отключить оптимизации (для отладки)
go build -gcflags="-N -l" main.go
```

### Примеры вывода

**Inlining и escape analysis (`-m`):**

```bash
$ go build -gcflags="-m" main.go

./main.go:5:6: can inline add
./main.go:10:12: inlining call to add
./main.go:15:6: create() leaks &x to heap
./main.go:20:6: compute() does not escape
```

**Escape analysis (`-d=esc`):**

```bash
$ go build -gcflags="-d=esc" main.go

./main.go:15:6: &x escapes to heap:
./main.go:15:6:   return &x
./main.go:15:6:   return &x: move to heap
```

**SSA вывод (`-S`):**

```bash
$ go build -gcflags="-S" main.go 2>&1 | head -30

"".add(SB), $0-24
0x0000  00000  <unknown>  MOVQ  a+0(FP), AX
0x0003  00003  <unknown>  MOVQ  b+8(FP), BX
0x0008  00008  <unknown>  ADDQ  BX, AX
0x000b  00011  <unknown>  MOVQ  AX, ret+16(FP)
0x000f  00015  <unknown>  RET
```

### Внутренние флаги `-d=...`

Флаг `-d` открывает доступ к внутренним отладочным настройкам компилятора:

| Флаг | Описание |
|------|----------|
| `-d=esc` | Детальный вывод escape analysis |
| `-d=checkptr` | Проверка указателей на безопасность |
| `-d=softfloat` | Эмуляция float point (для платформ без FPU) |
| `-d=gccheckmark` | Проверка сборщика мусора |
| `-d=lfstack` | Отладка lock-free стека |
| `-d=panic` | Дополнительная информация о панике |
| `-d=sym` | Отладка символов |
| `-d=help` | Показать все доступные `-d` флаги |

## 🔧 Ассемблер (go tool asm): asmflags

**Ассемблер Go** (`go tool asm`) превращает файлы `.s` в объектные файлы. Флаг `-asmflags` передаёт параметры ассемблеру.

### Важное уточнение

Флаг `-asmflags` работает **только с `.s` файлами** (ассемблерный код). Для обычных `.go` файлов нужно использовать `-gcflags="-S"`.

**Правильный способ посмотреть ассемблерный код Go программы:**

```bash
# Через компилятор (для .go файлов)
go build -gcflags="-S" main.go
```

### Как посмотреть все флаги ассемблера

```bash
go tool asm -help
```

**Популярные флаги:**

| Флаг | Описание | Пример |
|------|----------|--------|
| `-S` | Показать ассемблерный и машинный код | `-S` (только для .s файлов) |
| `-D identifier=value` | Определить символ | `-D VERSION=1` |
| `-I directory` | Добавить директорию для include | `-I ./asm` |
| `-d=...` | Внутренние отладочные флаги | `-d=help` |

### Ассемблер в Go: зачем он нужен

**Ассемблер в Go** — это не «настоящий» ассемблер, а **Plan 9 assembly**. Он используется для написания низкоуровневого кода, который требует максимального контроля над процессором.

**Когда используется:**

1. **Runtime Go:** Планировщик, GC, работа с горутинами
2. **Криптография:** AES, SHA, SHA256 — критично к производительности
3. **Оптимизации:** Когда компилятор не может сгенерировать оптимальный код
4. **Системные вызовы:** Прямой доступ к syscall

**Пример из стандартной библиотеки:**

```go
// crypto/aes/aes_amd64.go
func aesEncrypt(dst, src []byte, keys *uint32)

// crypto/aes/aes_amd64.s
TEXT ·aesEncrypt(SB), NOSPLIT, $0-32
    MOVQ    dst+0(FP), AX
    MOVQ    src+8(FP), BX
    // ... AES инструкции
    RET
```

**Синтаксис Plan 9:**

```
MOVQ    AX, BX      // Переместить AX в BX (Q = quadword, 64 бита)
ADDQ    BX, AX      // Добавить BX к AX
XORL    AX, AX      // XOR AX с самим собой (обнуление)
CALL    runtime·gc(SB)  // Вызов функции
RET                 // Возврат
```

**📚 Официальная документация:**

Полное описание синтаксиса и инструкций — в [мануале Plan 9 Assembly](https://9p.io/sys/doc/asm.html). Это оригинальный документ от создателей Go, описывающий ассемблер, который используется в языке.

**Обозначения суффиксов:**

| Суффикс | Размер | Описание |
|---------|--------|----------|
| `B` | 1 байт | Byte |
| `W` | 2 байта | Word |
| `L` | 4 байта | Long |
| `Q` | 8 байт | Quadword |

**Когда НЕ нужен:**

- Обычный бизнес-код — компилятор справляется отлично
- Веб-сервисы, API — разница незаметна
- Прототипы — трать время на логику, а не оптимизации

**Когда стоит рассмотреть:**

- Криптография (AES, SHA)
- Обработка больших объёмов данных
- Hot path в производительном коде
- Работа с конкретными инструкциями CPU (AVX, SSE)

## 🔖 Build Tags: условная компиляция

**Build tags** (или build constraints) — это механизм, который позволяет компилятору включать или исключать файлы из сборки в зависимости от условий.

Это не просто «фича» — это фундаментальный инструмент для написания кроссплатформенного кода. Когда ты видишь, что Go работает на Linux, Windows, macOS, ARM, AMD64 — всё это благодаря build tags.

### Как это работает

Build tags указываются в **первой строке файла** (или после строки с комментарием `//go:build`):

```go
//go:build linux && amd64

package platform
```

Файл будет скомпилирован **только** если оба условия истинны: ОС — Linux и архитектура — amd64.

### Синтаксис условий

```go
//go:build windows           // только Windows
//go:build darwin            // только macOS  
//go:build linux             // только Linux
//go:build !windows          // не Windows
//go:build linux || darwin   // Linux или macOS
//go:build go1.21            // версия Go 1.21+
//go:build go1.21 && linux   // Go 1.21+ и Linux
//go:build unix              // любая Unix-система
```

**Логические операторы:**
- `&&` — И (оба условия должны быть истинны)
- `||` — ИЛИ (хотя бы одно истинно)
- `!` — НЕ (инверсия)
- Пробел между условиями означает И: `//go:build linux amd64` = `linux && amd64`

### Специальные теги

Go автоматически определяет следующие теги:

| Тег | Значение |
|-----|----------|
| `GOOS` | Операционная система: `linux`, `windows`, `darwin`, `freebsd` |
| `GOARCH` | Архитектура: `amd64`, `arm64`, `386`, `ppc64le` |
| `GOOS_GOARCH` | Комбинация: `linux_amd64`, `darwin_arm64` |
| `unix` | Любая Unix-подобная система |
| `go1.X` | Версия Go: `go1.18`, `go1.19`, и т.д. |

### Практическое применение

**1. Платформенно-зависимый код:**

```go
// socket_linux.go
//go:build linux

package network

func NewSocket() net.Conn {
    // Linux-specific реализация через epoll
}

// socket_darwin.go
//go:build darwin

package network

func NewSocket() net.Conn {
    // macOS-specific реализация через kqueue
}

// socket_default.go
//go:build !linux && !darwin

package network

func NewSocket() net.Conn {
    // реализация по умолчанию
}
```

Компилятор сам выберет нужный файл в зависимости от целевой платформы.

**2. Версионные ограничения:**

```go
//go:build go1.21

package optimized

// Используем новые функции Go 1.21
func Fast() {
    // новый API
}
```

**3. Отладочный код:**

```go
//go:build debug

package main

func debugLog(msg string) {
    fmt.Println("DEBUG:", msg)
}

// В обычном режиме этот файл не компилируется
```

Сборка с тегом:

```bash
go build -tags debug
```

**4. Экспериментальные функции:**

```go
//go:build experimental

package feature

func NewFeature() {
    // ещё не готово для продакшена
}
```

### Старый формат (до Go 1.17)

В старых проектах ты можешь встретить:

```go
// +build linux,amd64
```

В старом формате:
- Запятая означала **И**: `linux,amd64` = `linux && amd64`
- Пробел означал **ИЛИ**: `linux darwin` = `linux || darwin`
- `!` — НЕ: `!windows`

**Go 1.17+ использует новый формат `//go:build`**, но старый всё ещё поддерживается для совместимости.

### Как посмотреть определённые теги

```bash
go env GOOS GOARCH
# darwin
# arm64

go list -f '{{.GoFiles}}' ./...
# покажет файлы, включённые в сборку
```

### Частые ошибки

**1. Build tag не в первой строке:**

```go
// ❌ Неправильно
package platform

//go:build linux  // игнорируется!

// ✅ Правильно
//go:build linux

package platform
```

**2. Пустая строка перед тегом:**

```go
// ❌ Неправильно (пустая строка перед тегом)

//go:build linux
package platform

// ✅ Правильно
//go:build linux

package platform
```

**3. Конфликтующие условия:**

```go
//go:build linux && !linux  // никогда не выполнится!

package impossible
```

Такой файл никогда не будет скомпилирован.

## 🔗 Linker: линковка и ldflags

После того как компилятор превратил каждый пакет в `.a`-файл, в дело вступает **линкер**. Его задача — собрать все объектные файлы в единый бинарник.

### Что делает линкер

```
┌─────────────────────────────────────────────────────────────┐
│  .a-файлы пакетов                                           │
│  ↓                                                          │
│  1. Разрешение внешних символов (функции, переменные)       │
│  ↓                                                          │
│  2. Объединение секций (код, данные, отладка)               │
│  ↓                                                          │
│  3. Генерация таблицы символов                              │
│  ↓                                                          │
│  4. Применение флагов (-ldflags)                            │
│  ↓                                                          │
│  5. Создание исполняемого файла                             │
└─────────────────────────────────────────────────────────────┘
```

### Флаги линковщика: полный список

Все флаги можно посмотреть через `go tool link -help`, но большинство из них тебе не понадобятся. Вот только те, что имеют практическую ценность:

| Флаг | Зачем нужен | Когда использовать |
|------|-------------|-------------------|
| `-s` | Удалить таблицу символов | Продакшен, уменьшение размера |
| `-w` | Удалить отладку (DWARF) | Продакшен, уменьшение размера |
| `-X pkg.var=value` | Вшить переменную в бинарник | Версия, время сборки |
| `-E entry` | Сменить точку входа | Отладка, эксперименты |
| `-H=windowsgui` | Без консольного окна | Windows GUI приложения |
| `-extldflags '-static'` | Статическая линковка | Переносимые бинарники |
| `-race` | Race detector | Отладка конкурентности |

Остальные флаги вроде `-D`, `-T`, `-R`, `-asan`, `-msan` нужны для узкоспециальных задач — если столкнёшься, документация расскажет подробнее.

---

### Удаление отладочной информации: `-s -w`

```bash
go build -ldflags="-s -w" -o app
```

Два флага, которые всегда идут вместе:

- `-s` (strip) — удаляет таблицу символов
- `-w` — удаляет DWARF-отладку

**Что это даёт:**

```bash
# Обычная сборка
go build -o app
ls -lh app
# 8.5M

# Без отладки
go build -ldflags="-s -w" -o app
ls -lh app
# 5.2M
```

Разница может достигать 30-40%. Для продакшена это стандарт — отладка там не нужна.

**Что ты теряешь:**

- Невозможно использовать отладчик (delve)
- Stack trace будет без имён функций и номеров строк
- `go tool nm` и `go tool objdump` покажут меньше информации

**Вывод:** для продакшена — однозначно да. Для отладки — оставляй как есть.

---

### Вшивание переменных: `-X pkg.var=value`

```go
// main.go
package main

import "fmt"

var Version = "unknown"
var BuildTime = "unknown"

func main() {
    fmt.Printf("Version: %s, Build: %s\n", Version, BuildTime)
}
```

```bash
VERSION="1.2.3"
BUILD=$(date -u +%Y-%m-%d)

go build -ldflags="-X main.Version=$VERSION -X main.BuildTime=$BUILD" -o app
```

**Зачем:** версия и время сборки «вшиваются» прямо в бинарник. Не нужно парсить файлы или использовать环境变量.

**Важно:** работают только строковые переменные. Для чисел придётся парсить:

```go
var BuildNumber int  // ❌ не сработает напрямую

// Парсинг в коде:
BuildNumber, _ = strconv.Atoi(buildNumberStr)
```

---

### Смена точки входа: `-E entry`

```bash
go build -ldflags="-E _start"
```

**Важно:** заглавная `-E`, а не строчная `-e`.

Флаг меняет точку входа с `runtime.main` на указанную.

**Когда может понадобиться:**

- Отладка инициализации Go
- Кастомные обёртки вокруг `main()`

**Пример:**

```go
package main

import "fmt"

func _start() {
    fmt.Println("Custom entry point")
}
```

---

### Статическая линковка: `-extldflags '-static'`

```bash
go build -ldflags="-extldflags '-static'"
```

Все библиотеки линкуются статически — бинарник не зависит от системных библиотек.

**Важно:** для полной статической линковки нужен `CGO_ENABLED=1` и статические библиотеки в системе.

---

### Таблица символов

Таблица символов содержит имена всех функций и переменных в бинарнике. Посмотреть её можно так:

```bash
go tool nm mybinary | head -20
```

Пример вывода:

```
00000000004a5a40 T runtime.main
00000000004a5b20 T main.main
00000000005f0a00 D main.Version
00000000005f0a08 B myapp.Buffer
```

**Типы символов:**

| Символ | Что означает | Пример |
|--------|--------------|--------|
| `T`, `t` | Текстовая секция (код) | `T runtime.main`, `t helperFunc` |
| `D`, `d` | Секция данных (инициализированные переменные) | `D main.Version`, `d config` |
| `B`, `b` | BSS (неинициализированные переменные) | `B myapp.Buffer`, `b uninitialized` |
| `R`, `r` | Read-only (константы) | `R main.Pi` |
| `U` | Undefined (внешний символ) | `U printf` |

**Заглавные vs строчные:**

- **Заглавные** (`T`, `D`, `B`) — экспортируемые символы
- **Строчные** (`t`, `d`, `b`) — локальные (неэкспортируемые)

**Подробнее о BSS:**

BSS (Block Started by Symbol) — это секция для переменных, которые не были явно инициализированы. Они автоматически обнуляются при запуске программы.

```go
package main

var Initialized = 42    // D (Data) — инициализирована
var Uninitialized int   // B (BSS) — неинициализирована, будет 0
```

**BSS vs Data:**

```
┌─────────────────────────────────────────────────────────────┐
│  Data (инициализированные):                                 │
│  var Version = "1.0.0"  → занимает место в бинарнике        │
│                                                             │
│  BSS (неинициализированные):                                │
│  var Buffer [1024]byte  → не занимает места в бинарнике     │
│                         → обнуляется при запуске            │
└─────────────────────────────────────────────────────────────┘
```

**Зачем это знать:**

- **Отладка:** понять, где находится символ
- **Оптимизация:** BSS не увеличивает размер бинарника
- **Безопасность:** U (undefined) может указывать на недостающую библиотеку

## 🧠 SSA: промежуточное представление

**SSA (Static Single Assignment)** — это промежуточное представление кода, которое использует компилятор Go для оптимизаций.

### Что такое SSA

В SSA каждая переменная присваивается **только один раз**. Если в исходном коде переменная изменяется, в SSA создаётся новая версия:

```go
// Исходный код
x := 1
x = x + 1
x = x * 2

// SSA (упрощённо)
x1 = 1
x2 = x1 + 1
x3 = x2 * 2
```

Это может показаться избыточным, но SSA упрощает анализ и оптимизацию кода.

### Как посмотреть SSA

```bash
GOSSAFUNC=main go build
# Создаст ssa.html с визуализацией
```

Или для конкретного пакета:

```bash
go build -gcflags="-S -l" ./...  # вывод SSA в консоль
```

### 📋 Детальное описание этапов

#### 1. sources

Исходный код функции. На этом этапе компилятор просто читает твой `.go`-файл.

```go
func add(a, b int) int {
    return a + b
}
```

**Что искать в ssa.html:** исходные строки кода, номера строк для отладки.

---

#### 2. ast

**AST (Abstract Syntax Tree)** — абстрактное синтаксическое дерево.

Компилятор парсит исходный код и строит дерево узлов:

```
FuncDecl: add
├── Params: a, b (int)
├── Results: int
└── Body:
    Return
    └── BinaryExpr: +
        ├── a
        └── b
```

**Что искать:** как твой код представлен в виде дерева. Ошибки парсинга видны здесь.

---

#### 3. before insert phis

Перед вставкой **φ-функций** (phi-функций).

φ-функции — это специальный механизм SSA для обработки ветвлений. Они «выбирают» правильное значение переменной в зависимости от того, по какому пути выполнению пришёл код.

```go
if condition {
    x = 1
} else {
    x = 2
}
// Здесь нужна φ-функция: x = φ(1, 2)
```

**Что искать:** как код выглядит до обработки циклов и условий.

---

#### 4. start

Начало SSA-представления. Код преобразован в форму SSA, где каждая переменная присваивается один раз.

```go
// Исходный код
x := 1
x = x + 1

// SSA
v1 = IntConst 1
v2 = Add v1 1
```

**Что искать:** базовые блоки (blocks), значения (values), типы операций.

---

#### 5. number lines

Нумерация строк для отладки. Компилятор связывает каждую SSA-операцию с номером строки в исходном коде.

**Зачем нужно:** для отладчика и сообщений об ошибках.

---

#### 6. early phielim and copyelim

**Раннее удаление φ-функций и копий.**

Если φ-функция избыточна (все аргументы одинаковы), она удаляется. Копии вида `a = b` тоже удаляются, если можно использовать `b` напрямую.

```go
// До
v3 = φ(v1, v1)  // все аргументы одинаковы
v4 = Copy v3

// После
v4 = v1
```

**Что искать:** какие φ-функции и копии удалены на раннем этапе.

---

#### 7. early deadcode

**Раннее удаление мёртвого кода.**

Код, результат которого не используется, удаляется сразу.

```go
// До
v1 = Add a b      // результат не используется
v2 = Mul c d
return v2

// После
v2 = Mul c d
return v2
```

**Что искать:** какие вычисления оказались ненужными.

---

#### 8. short circuit

**Оптимизация короткого замыкания.**

Логические операторы `&&` и `||` оптимизируются:

```go
// До
if a && b { ... }

// SSA
if a {
    if b { ... }
}

// После (если a ложно, b не вычисляется)
```

**Что искать:** как условия преобразуются в цепочки проверок.

---

#### 9. decompose user + pre-opt deadcode

**Декомпозиция пользовательских типов + удаление мёртвого кода перед оптимизацией.**

Структуры и массивы раскладываются на отдельные поля/элементы:

```go
// До
type Point struct{ X, Y int }
p := Point{1, 2}

// После
v1 = IntConst 1  // p.X
v2 = IntConst 2  // p.Y
```

**Что искать:** как твои типы представляются на низком уровне.

---

#### 10. opt

**Основная оптимизация.**

Набор общих оптимизаций:
- Упрощение арифметики
- Удаление избыточных операций
- Свёртка констант

```go
// До
v1 = Add a 0
v2 = Mul v1 1

// После
v2 = a
```

---

#### 11. zero arg cse

**Common Subexpression Elimination для функций с нулём аргументов.**

Если функция без аргументов вызывается несколько раз и не имеет побочных эффектов, результат кэшируется:

```go
// До
v1 = Call len()
v2 = Call len()

// После
v1 = Call len()
v2 = v1
```

---

#### 12. opt deadcode

**Удаление мёртвого кода после оптимизации.**

Повторная проверка на неиспользуемые вычисления.

---

#### 13. generic cse

**Generic Common Subexpression Elimination.**

Повторное использование результатов одинаковых вычислений:

```go
// До
v1 = Add a b
v2 = Add a b

// После
v1 = Add a b
v2 = v1
```

**Что искать:** какие вычисления были объединены.

---

#### 14. phiopt + gcse deadcode

**Оптимизация φ-функций + глобальное CSE.**

φ-функции упрощаются, общие подвыражения находятся глобально (между блоками).

---

#### 15. nilcheckelim + prove

**Удаление проверок nil + доказательство условий.**

Компилятор доказывает, что некоторые проверки избыточны:

```go
// До
if p != nil {
    x = *p  // проверка nil уже была
}

// После (проверка удалена)
x = *p
```

**`prove`** — анализатор, который доказывает безопасность удаления проверок.

**Что искать:** какие проверки границ и nil были удалены.

---

#### 16. divisible + divmod + middle opt + early fuse

**Оптимизация деления:**

- **divisible** — проверка делимости без остатка
- **divmod** — объединение деления и остатка
- **middle opt** — средние оптимизации
- **early fuse** — слияние операций

```go
// До
q = a / b
r = a % b

// После (одна операция)
q, r = DivMod a b
```

---

#### 17. expand calls

**Раскрытие вызовов функций.**

Вызовы функций преобразуются в низкоуровневые операции:

```go
// До
result = someFunc(a, b)

// После
// Подготовка аргументов, вызов, получение результата
```

---

#### 18. decompose builtin

**Декомпозиция встроенных функций.**

`len`, `cap`, `append` и другие встроенные функции раскладываются на примитивные операции.

---

#### 19. softfloat + branchelim + late opt

- **softfloat** — эмуляция float point для платформ без FPU
- **branchelim** — удаление избыточных ветвлений
- **late opt** — поздние оптимизации

```go
// До
if x > 0 {
    return true
} else {
    return false
}

// После (branchelim)
return x > 0
```

---

#### 20. dead auto elim + sccp + generic deadcode

- **dead auto elim** — удаление автоматических переменных, которые не читаются
- **sccp** — Sparse Conditional Constant Propagation (распространение констант)
- **generic deadcode** — общее удаление мёртвого кода

**SCCP** находит переменные, которые всегда имеют одно значение:

```go
// До
x := 5
if x == 5 {  // всегда true
    ...
}

// После (sccp)
// Ветка if выполняется всегда
```

---

#### 21. late fuse + check bce + dse

- **late fuse** — позднее слияние операций
- **check bce** — проверка Boundary Check Elimination
- **dse** — Dead Store Elimination (удаление мёртвых записей)

**DSE** удаляет записи в память, которые перезаписываются:

```go
// До
arr[i] = 1
arr[i] = 2  // первая запись мёртвая

// После
arr[i] = 2
```

**BCE** удаляет проверки границ массива:

```go
// До
for i := 0; i < len(arr); i++ {
    _ = arr[i]  // проверка: i < len(arr)
}

// После (если компилятор доказал безопасность)
_ = arr[i]  // без проверки
```

---

#### 22. memcombine + writebarrier + lower

- **memcombine** — объединение операций памяти
- **writebarrier** — барьеры записи для сборщика мусора
- **lower** — понижение до машинных инструкций

**Write barrier** нужен для GC:

```go
// Перед записью указателя
writeBarrier()
*ptr = newValue
```

---

#### 23. addressing modes + late lower + pair

- **addressing modes** — выбор режима адресации (регистр, память, смещение)
- **late lower** — позднее понижение сложных операций
- **pair** — оптимизация пар значений (например, результат деления: частное + остаток)

---

#### 24-25. lowered deadcode + lowered cse

**Удаление мёртвого кода и CSE после понижения.**

Повторная оптимизация уже на низком уровне.

**Время выполнения:** ~1708 ns (указано в ssa.html).

---

#### 26. elim unread autos

**Удаление непрочитанных автоматических переменных.**

Если переменная создана, но никогда не читается — удаляется.

---

#### 27. tighten tuple selectors

**Уплотнение селекторов кортежей.**

Кортежи (множественные возвращаемые значения) оптимизируются:

```go
// До
result, ok := someFunc()
v = result  // селектор кортежа

// После (если возможно)
v, ok = someFunc()
```

---

#### 28-32. Поздние оптимизации

- **lowered deadcode** — мёртвый код после понижения
- **checkLower** — проверка корректности понижения
- **late phielim and copyelim** — позднее удаление φ-функций и копий
- **tighten** — финальное уплотнение кода
- **late deadcode** — финальное удаление мёртвого кода

**Время выполнения** указано в ssa.html для каждого этапа.

#### 33-35. critical + phi tighten + likelyadjust

- **critical** — обработка критических путей выполнения
- **phi tighten** — уплотнение φ-функций
- **likelyadjust** — корректировка предсказаний ветвлений (branch prediction hints)

---

#### 36. layout

**Размещение кода.**

Базовые блоки располагаются в памяти оптимальным образом для производительности.

---

#### 37. schedule

**Планирование инструкций.**

Инструкции переупорядочиваются для лучшего использования конвейера процессора:

```
// До
load r1, [addr1]
load r2, [addr2]
add r3, r1, r2

// После (если load независимы)
load r1, [addr1]
load r2, [addr2]  // параллельно
add r3, r1, r2
```

**Самый долгий этап** (~6375 ns).

---

#### 38. late nilcheck + flagalloc + regalloc

- **late nilcheck** — финальные проверки nil
- **flagalloc** — распределение флагов процессора
- **regalloc** — распределение регистров

**Regalloc** — критический этап. Переменные распределяются по регистрам процессора. Если регистров не хватает, переменные «выливаются» в память (spill).

---

#### 39. loop rotate + trim

- **loop rotate** — вращение циклов для оптимизации ветвлений
- **trim** — обрезка избыточного кода

**Loop rotate** преобразует цикл:

```go
// До
for i := 0; i < n; i++ {
    body()
}

// После (вращение)
i := 0

    body()
    i++
}
```

## ⚡ Inlining: встраивание функций

**Inlining** — это оптимизация, при которой вызов функции заменяется её телом.

### Как работает inlining

```go
// Исходный код
func add(a, b int) int {
    return a + b
}

result := add(x, y)

// После inlining
result := x + y  // вызов функции исчез
```

**Преимущества:**
- Нет накладных расходов на вызов функции
- Открывает возможности для дальнейших оптимизаций

**Недостатки:**
- Увеличивает размер кода
- Может замедлить компиляцию

### Когда Go встраивает функции

Go автоматически встраивает **маленькие функции**. Критерии меняются от версии к версии, но примерно так:

```go
// ✅ Будет встроена (простая функция)
func add(a, b int) int {
    return a + b
}

// ❌ Не будет встроена (сложная функция)
func complex() int {
    x := 0
    for i := 0; i < 100; i++ {
        x += i
    }
    return x
}
```

### Как проверить inlining

```bash
go build -gcflags="-m"  # показывает решения об inlining
```

Пример вывода:

```
./main.go:5:6: can inline add
./main.go:10:12: inlining call to add
```

### Запрет inlining

Иногда нужно запретить встраивание (для профилирования, например):

```go
//go:noinline
func benchmarkMe() {
    // эта функция не будет встроена
}
```

Или через флаг:

```bash
go build -gcflags="-l"  # запретить все inlining
```

## 🔍 Escape Analysis: анализ утечек

**Escape Analysis** — это процесс, при котором компилятор определяет, где будет выделена память: на стеке или в куче.

### Стек vs Куча

```
┌─────────────────────────────────────────────────────────┐
│  Стек (Stack)          Куча (Heap)                      │
│  ─────────────────────────────────────────────────────  │
│  Быстро                  Медленнее                      │
│  Автоматическая очистку  Требуется GC                   │
│  Локальные переменные    "Убежавшие" переменные         │
└─────────────────────────────────────────────────────────┘
```

**Стек** — быстрая память для локальных переменных. Очищается автоматически при выходе из функции.

**Куча** — медленнее, требует работы сборщика мусора. Но переменные в куче живут дольше.

### Когда переменная «убегает» в кучу

```go
// ❌ Убегает: возвращается указатель
func create() *int {
    x := 42
    return &x  // x должен жить после выхода из функции
}

// ✅ Остаётся на стеке: не возвращается
func compute() int {
    x := 42
    return x  // x копируется, оригинал не нужен
}
```

**Правила escape analysis (упрощённо):**

1. Если указатель на переменную возвращается из функции — переменная убегает в кучу
2. Если указатель сохраняется в глобальной переменной — убегает
3. Если переменная слишком большая — может убежать
4. Если тип переменной неизвестен на этапе компиляции (interface{}) — убегает

### Как посмотреть escape analysis

```bash
go build -gcflags="-m"  # показывает escape analysis
```

Пример вывода:

```
./main.go:5:6: create() leaks &x to heap
./main.go:10:6: compute() does not escape
```

Более детальный вывод:

```bash
go build -gcflags="-m -m"  # больше деталей
```

### Оптимизация с учётом escape analysis

```go
// ❌ Плохо: утечка в кучу
func process() *[]int {
    data := make([]int, 1000)
    return &data
}

// ✅ Лучше: возврат значения
func process() []int {
    data := make([]int, 1000)
    return data  // срез копируется, данные могут остаться на стеке
}
```

### Подсказка компилятору: `//go:noescape`

Для функций на ассемблере можно указать, что указатели не убегают:

```go
//go:noescape
func myFunc(ptr uintptr) {
    // ptr не убежит из функции
}
```

Это позволяет компилятору выделить память на стеке вместо кучи.

## 🗑️ GC Internals: сборщик мусора

**Сборщик мусора (Garbage Collector, GC)** в Go — это один из самых важных компонентов runtime. Он автоматически освобождает память, которую программа больше не использует.

Понимание того, как работает GC, критично для написания производительных программ с предсказуемым поведением.

### Tri-color Mark-and-Sweep

Go использует алгоритм **tri-color mark-and-sweep** (трёхцветная пометка и очистка). Это не просто «сборка мусора» — это точный алгоритм, который работает параллельно с твоей программой.

**Как это работает:**

Представь, что все объекты в куче — это клетки на поле. Каждая клетка может быть одного из трёх цветов:

```
┌─────────────────────────────────────────────────────────────┐
│  🤍 Белый    →  «Возможно мусор» (не проверен)              │
│  🖤 Чёрный   →  «Точно жив» (проверен, достижим)            │
│  🩶 Серый    →  «На проверке» (сам достижим, дети — нет)    │
└─────────────────────────────────────────────────────────────┘
```

**Важно:** «Дети» (дочерние объекты) — это не наследование! Это объекты, на которые есть **ссылки**. Если объект A содержит поле типа `*B`, то B — ребёнок A.

```go
type A struct {
    B *B  // B — дочерний объект A (A ссылается на B)
}
```

**Процесс сборки:**

```
┌─────────────────────────────────────────────────────────────┐
│  1. Все объекты белые (не проверены)                        │
│     ↓                                                       │
│  2. Корневые объекты (стек, глобальные) становятся серыми   │
│     ↓                                                       │
│  3. Пока есть серые объекты:                                │
│     - Берём серый объект                                    │
│     - Помечаем его детей как серых                          │
│     - Делаем объект чёрным                                  │
│     ↓                                                       │
│  4. Все белые объекты — мусор, удаляем                      │
└─────────────────────────────────────────────────────────────┘
```

**Визуализация:**

```
Начало (все объекты белые):
  [🤍 Root] → [🤍 A]      # Root ссылается на A (A — ребёнок Root)
     │
  [🤍 B]                  # B не связан (никто на него не ссылается)

Шаг 1 (Root стал серым — начал проверку):
  [🩶 Root] → [🤍 A]      # Root проверяется, A ещё белый
     │
  [🤍 B]

Шаг 2 (Root чёрный, A серый — проверяется):
  [🖤 Root] → [🩶 A]      # Root жив, A проверяется
     │
  [🤍 B]

Шаг 3 (все достижимые чёрные, B — белый мусор):
  [🖤 Root] → [🖤 A]      # Root и A — живые (чёрные)
     │
  [🤍 B] ← УДАЛИТЬ        # B — белый (недостижим, никто не ссылается)
```

**Простыми словами:**

```
┌─────────────────────────────────────────────────────────────┐
│  🤍 Белый = «Не знаю, жив ли ты» → если остался → мусор     │
│  🩶 Серый = «Проверяю тебя и твои связи»                    │
│  🖤 Чёрный = «Ты точно жив, не трогаю»                      │
└─────────────────────────────────────────────────────────────┘
```

**«Дети» = ссылки, а не наследование:**

```go
type User struct {
    Profile *Profile  // Profile — ребёнок User (ссылка)
    Orders  []Order   // Order — ребёнок User (слайс ссылок)
}
// GC пройдёт: User → Profile и User → Orders
```

### Write Barrier

**Write barrier** — это механизм, который позволяет GC работать **параллельно** с программой, без остановки мира (stop-the-world).

**Проблема:** если программа изменяет указатели во время сборки мусора, GC может пропустить живой объект.

**Решение:** перед каждой записью указателя компилятор вставляет вызов write barrier:

```go
// До
ptr.field = newValue

// После (компилятор вставляет)
writeBarrier()
ptr.field = newValue
```

**Что делает write barrier:**

```
┌─────────────────────────────────────────────────────────────┐
│  writeBarrier():                                            │
│  1. Если newValue белый (не проверен)                       │
│  2. → пометить newValue как серый                           │
│  3. → GC проверит newValue позже                            │
└─────────────────────────────────────────────────────────────┘
```

Это гарантирует, что ни один живой объект не будет удалён, даже если программа изменяет указатели во время сборки.

### Фазы работы GC

Сборка мусора происходит в несколько фаз:

```
┌─────────────────────────────────────────────────────────────┐
│  Фаза 1: Mark Start (STW ~0.1ms)                            │
│  - Короткая остановка мира                                  │
│  - Пометить корневые объекты как серые                      │
│  ↓                                                          │
│  Фаза 2: Mark (параллельно, ~1-10ms)                        │
│  - Программа работает                                       │
│  - GC помечает достижимые объекты                           │
│  - Write barrier защищает от изменений                      │
│  ↓                                                          │
│  Фаза 3: Mark Terminate (STW ~0.1ms)                        │
│  - Короткая остановка                                       │
│  - Завершение пометки                                       │
│  ↓                                                          │
│  Фаза 4: Sweep (параллельно, ~1-10ms)                       │
│  - Программа работает                                       │
│  - Освобождение белых объектов                              │
└─────────────────────────────────────────────────────────────┘
```

**STW (Stop-The-World)** — это пауза, когда программа останавливается. В Go 1.5+ STW минимизирован до ~0.1-1ms.

### Настройка GC: GOGC и GOMEMLIMIT

**GOGC** — переменная окружения, которая управляет частотой сборки мусора.

```bash
GOGC=100  # по умолчанию: GC запускается, когда куча удваивается
GOGC=50   # GC запускается чаще (агрессивная сборка)
GOGC=200  # GC запускается реже (меньше пауз, больше памяти)
GOGC=off  # отключить GC (только для специальных случаев!)
```

**Как работает GOGC:**

```
┌─────────────────────────────────────────────────────────────┐
│  GOGC=100 (по умолчанию)                                    │
│                                                             │
│  После GC: куча = 100MB                                     │
│  Следующая GC: когда куча = 200MB (100MB + 100%)            │
│                                                             │
│  GOGC=50                                                    │
│  После GC: куча = 100MB                                     │
│  Следующая GC: когда куча = 150MB (100MB + 50%)             │
│                                                             │
│  GOGC=200                                                   │
│  После GC: куча = 100MB                                     │
│  Следующая GC: когда куча = 300MB (100MB + 200%)            │
└─────────────────────────────────────────────────────────────┘
```

**GOMEMLIMIT** (Go 1.19+) — жёсткий лимит памяти для кучи.

```bash
GOMEMLIMIT=512MiB  # GC будет агрессивно работать, чтобы не превысить лимит
```

В отличие от GOGC, GOMEMLIMIT — это жёсткое ограничение. Если программа приближается к лимиту, GC запускается чаще.

### Мониторинг GC

**Статистика GC:**

```go
package main

import (
    "fmt"
    "runtime"
)

func main() {
    var m runtime.MemStats
    runtime.ReadMemStats(&m)
    
    fmt.Printf("Alloc = %v KB", m.Alloc/1024)
    fmt.Printf("\tTotalAlloc = %v KB", m.TotalAlloc/1024)
    fmt.Printf("\tSys = %v KB", m.Sys/1024)
    fmt.Printf("\tNumGC = %v\n", m.NumGC)
    fmt.Printf("\tPauseTotalNs = %v ms\n", m.PauseTotalNs/1e6)
}
```

**Поля MemStats:**

| Поле | Описание |
|------|----------|
| `Alloc` | Текущее использование кучи |
| `TotalAlloc` | Всего выделено (растёт) |
| `Sys` | Всего памяти запрошено у ОС |
| `NumGC` | Количество сборок мусора |
| `PauseTotalNs` | Общее время пауз STW |
| `PauseNs` | Время каждой паузы (циклический буфер) |
| `GCCPUFraction` | Доля CPU, используемая GC |

**GC trace:**

```bash
GODEBUG=gctrace=1 go run main.go
```

Пример вывода:

```
gc 1 @0.012s 0%: 0.11+0.20+0.015 ms clock, 0.69+0.11/0.41/0.82+0.090 MB, 6 → 4 → 2 MB
```

**Расшифровка:**
- `gc 1` — номер сборки
- `@0.012s` — время от старта программы
- `0%` — доля CPU
- `0.11+0.20+0.015 ms` — паузы (mark start + mark + mark terminate)
- `6 → 4 → 2 MB` — куча до GC → во время → после

### Оптимизация для GC

**1. Уменьши количество аллокаций:**

```go
// ❌ Плохо: аллокация на каждой итерации
for _, item := range items {
    result := process(item)  // новая аллокация
    use(result)
}

// ✅ Лучше: переиспользование буфера
var buf Buffer
for _, item := range items {
    buf.Reset()  // сброс без новой аллокации
    processInto(&buf, item)
    use(buf.Bytes())
}
```

**2. Используй sync.Pool:**

```go
var pool = sync.Pool{
    New: func() interface{} {
        return &Buffer{data: make([]byte, 1024)}
    },
}

func useBuffer() {
    buf := pool.Get().(*Buffer)
    defer pool.Put(buf)
    
    // работа с buf
}
```

**3. Избегай указателей в больших структурах:**

```go
// ❌ Плохо: много указателей → больше работы для GC
type Node struct {
    next *Node
    prev *Node
    data *Data
}

// ✅ Лучше: значения вместо указателей
type Node struct {
    nextID int
    prevID int
    data   Data
}
```

**4. Предварительное выделение (preallocate):**

```go
// ❌ Плохо: много реаллокаций
var result []int
for i := 0; i < 1000; i++ {
    result = append(result, i)
}

// ✅ Лучше: заранее известный размер
result := make([]int, 0, 1000)
for i := 0; i < 1000; i++ {
    result = append(result, i)
}
```

### Когда GC — проблема

**GC вызывает проблемы, если:**

1. **Много короткоживущих объектов** — GC работает постоянно
2. **Большие объекты** — долгая пометка и очистка
3. **Много указателей** — больше работы для write barrier
4. **Непредсказуемые аллокации** — паузы в неподходящее время

**Решение:**
- Профилируй с `GODEBUG=gctrace=1`
- Смотри `PauseTotalNs` и `GCCPUFraction`
- Уменьшай количество аллокаций
- Используй `sync.Pool` для частых объектов

## 🎯 Compiler Optimizations: оптимизации компилятора

Компилятор Go выполняет множество оптимизаций. Вот основные:

### Constant Folding

Вычисление константных выражений на этапе компиляции:

```go
x := 2 + 2 * 3  // компилятор вычислит: x := 8
```

### Dead Code Elimination

Удаление недостижимого кода:

```go
if false {
    // этот код будет удалён
}

func unreachable() {}  // будет удалена, если не вызывается
```

### Common Subexpression Elimination

Повторное использование результатов одинаковых вычислений:

```go
// До оптимизации
a := x * y + z
b := x * y + w

// После оптимизации
tmp := x * y
a := tmp + z
b := tmp + w
```

### Loop Invariant Code Motion

Вынос инвариантных вычислений из цикла:

```go
// До оптимизации
for i := 0; i < n; i++ {
    result := expensive() * i  // expensive() вызывается каждый раз
}

// После оптимизации
tmp := expensive()
for i := 0; i < n; i++ {
    result := tmp * i
}
```

### Bounds Check Elimination

Удаление проверок границ массива, когда компилятор может доказать их безопасность:

```go
// Если компилятор знает, что i < len(arr)
// проверка arr[i] может быть удалена
for i := 0; i < len(arr); i++ {
    _ = arr[i]  // проверка границы может быть удалена
}
```

## 📊 Binary Size Optimization: оптимизация размера бинарника

### Что влияет на размер

| Фактор | Влияние |
|--------|---------|
| Количество кода | Больше кода = больше бинарник |
| Встраивание функций | Увеличивает размер |
| Отладочная информация | +20-30% к размеру |
| Использование reflection | Увеличивает размер |
| Зависимости | Каждая зависимость добавляет код |

### Как уменьшить размер

**1. Удалить отладочную информацию:**

```bash
go build -ldflags="-s -w"
```

- `-s` — удалить таблицу символов
- `-w` — удалить DWARF (отладочную информацию)

**2. Использовать UPX (сжатие исполняемого файла):**

```bash
upx --best mybinary # На Mac в бете и работает криво
```

Может уменьшить размер в 2-3 раза, но добавляет накладные расходы на распаковку.

**3. Избегать ненужных зависимостей:**

```bash
go mod tidy  # удалить неиспользуемые
```

**4. Проверить, что внутри:**

```bash
go tool nm mybinary | head  # показать символы
```

### Пример уменьшения размера

```bash
# Обычная сборка
go build -o app
# app: 10MB

# Без отладочной информации
go build -ldflags="-s -w" -o app
# app: 7MB

# + UPX
upx --best app
# app: 3MB
```

## 📐 Memory Layout: расположение памяти

Когда Go-программа запускается, операционная система выделяет ей память. Эта память организована определённым образом, и понимание этой организации помогает писать более эффективный код.

### Структура памяти процесса

```
┌─────────────────────────────────────────────────────────────┐
│  Высокие адреса                                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                    Stack                            │    │
│  │                    (растёт вниз)                    │    │
│  │         ↓                                           │    │
│  │         │                                           │    │
│  │         │  свободная память                         │    │
│  │         │                                           │    │
│  │         ↑                                           │    │
│  │                    Heap                             │    │
│  │                    (растёт вверх)                   │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                    Data                             │    │
│  │                    (инициализированные данные)      │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                    BSS                              │    │
│  │                    (неинициализированные данные)    │    │
│  └─────────────────────────────────────────────────────┘    │
│  ┌─────────────────────────────────────────────────────┐    │
│  │                    Text                             │    │
│  │                    (код программы)                  │    │
│  └─────────────────────────────────────────────────────┘    │
│  Низкие адреса                                              │
└─────────────────────────────────────────────────────────────┘
```

### Секции бинарника

**Text (код):**
- Содержит машинный код программы
- Обычно доступна только для чтения
- Здесь живут функции, включая встроенные

**Data (данные):**
- Инициализированные глобальные переменные
- Статические данные с известными значениями

```go
var GlobalInt int = 42      // в Data
var GlobalSlice = []int{1, 2, 3}  // в Data
```

**BSS (Block Started by Symbol):**
- Неинициализированные глобальные переменные
- Переменные с нулевым значением по умолчанию

```go
var GlobalInt int           // в BSS (нулевое значение)
var GlobalSlice []int       // в BSS (nil)
```

**Heap (куча):**
- Динамически выделяемая память
- Переменные, которые «убежали» из стека (escape analysis)
- Управляется сборщиком мусора

**Stack (стек):**
- Локальные переменные функций
- Параметры вызова
- Адреса возврата
- Растёт и уменьшается автоматически

### Выравнивание (Memory Alignment)

Представь, что ты идёшь в библиотеку за книгами. Если книги стоят на полках подряд, ты берёшь их за один проход. А если каждая книга на разной полке — придётся бегать туда-сюда.

Примерно так же работает процессор с памятью.

**Выравнивание** — это требование процессора к тому, как данные располагаются в памяти. Процессор читает память не по одному байту, а блоками — обычно по 4 или 8 байт за раз.

**Почему это важно:** если данные выровнены по границе блока, чтение происходит за один цикл. Если нет — за два или больше.

```
┌─────────────────────────────────────────────────────────────┐
│  Выровнено (8 байт):                                        │
│  [0][1][2][3][4][5][6][7] ← один цикл чтения                │
│                                                             │
│  Не выровнено:                                              │
│  [0][1][2][3][4][5][6][7] ← первый цикл                     │
│           [8][9][A][B]    ← второй цикл (медленнее!)        │
└─────────────────────────────────────────────────────────────┘
```

### Правила выравнивания в Go

Каждый тип имеет своё **требование к выравниванию** (alignment). Это число показывает, по какой границе должен располагаться тип.

**Базовые типы:**

| Тип | Размер (байт) | Выравнивание (байт) | Примечание |
|-----|---------------|---------------------|------------|
| `bool` | 1 | 1 | |
| `int8`, `uint8`, `byte` | 1 | 1 | `byte` — алиас `uint8` |
| `int16`, `uint16` | 2 | 2 | |
| `int32`, `uint32`, `rune` | 4 | 4 | `rune` — алиас `int32` |
| `int64`, `uint64` | 8 | 8 | |
| `float32` | 4 | 4 | |
| `float64` | 8 | 8 | |
| `complex64` | 8 | 8 | `float32` + `float32` |
| `complex128` | 16 | 8 | `float64` + `float64` |

**Составные типы:**

| Тип | Размер (байт) | Выравнивание (байт) | Что внутри |
|-----|---------------|---------------------|------------|
| `uintptr`, указатели | 8 | 8 | Адрес в памяти |
| `string` | 16 | 8 | ptr (8) + len (8) |
| `[]T` (slice) | 24 | 8 | ptr (8) + len (8) + cap (8) |
| `map` | 8 | 8 | Указатель на hmap |
| `func` | 8 | 8 | Указатель на функцию |
| `interface` | 16 | 8 | type (8) + data (8) |
| `error` | 16 | 8 | Это интерфейс (ptr + data) |
| `chan` | 8 | 8 | Указатель на hchan |
| `struct` | Зависит от полей | Максимум среди полей | Сумма полей + padding |
| `array` | Зависит от типа | Как базовый тип | `len * size(элемента)` |

**Правило:** тип должен располагаться по адресу, который делится на его выравнивание без остатка.

```
int64 (выравнивание 8) должен быть по адресу: 0, 8, 16, 24, 32...
int32 (выравнивание 4) должен быть по адресу: 0, 4, 8, 12, 16...
int16 (выравнивание 2) должен быть по адресу: 0, 2, 4, 6, 8...
```

**📌 complex64 и complex128:**

```go
// complex64: два float32
var c64 complex64 = 1 + 2i  // 8 байт (4 + 4)

// complex128: два float64
var c128 complex128 = 1 + 2i  // 16 байт (8 + 8)
```

**📌 error — это интерфейс:**

```go
// error — это интерфейс из одного метода
type error interface {
    Error() string
}

// Занимает 16 байт (как любой интерфейс):
// - 8 байт: указатель на тип (type descriptor)
// - 8 байт: указатель на данные (value)
```

### Padding (выравнивающие отступы)

Компилятор добавляет невидимые байты между полями структуры для правильного выравнивания.

**Пример:**

```go
// ❌ Неоптимально: 32 байта
type Bad struct {
    a bool    // 1 байт + 7 padding (выравнивание под int64)
    b int64   // 8 байт
    c bool    // 1 байт + 7 padding (выравнивание под int64)
    d int64   // 8 байт
}
// Итого: 1+7+8+1+7+8 = 32 байта

// ✅ Оптимально: 24 байта
type Good struct {
    a int64   // 8 байт
    b int64   // 8 байт
    c bool    // 1 байт
    d bool    // 1 байт
              // 6 байт padding (выравнивание структуры по 8 байт)
}
// Итого: 8+8+1+1+6 = 24 байта
```

**Правило:** группируй поля по размеру от больших к меньшим.

**Как проверить размер и padding:**

```go
package main

import (
    "fmt"
    "unsafe"
)

type Bad struct {
    a bool
    b int64
    c bool
    d int64
}

type Good struct {
    a int64
    b int64
    c bool
    d bool
}

func main() {
    fmt.Println(unsafe.Sizeof(Bad{}))   // 32
    fmt.Println(unsafe.Sizeof(Good{}))  // 24
}
```

**Правило:** группируй поля по размеру от больших к меньшим.

### Layout структур в памяти

Структуры располагаются в памяти последовательно, с учётом выравнивания.

```go
type Person struct {
    Name string  // 16 байт (2 поля по 8: ptr + len)
    Age  int     // 8 байт
    ID   int32   // 4 байта
}
// Итого: 16 + 8 + 4 + 4 padding = 32 байта
// Выравнивание: 8 байт (по наибольшему базовому типу: uintptr/int)
```

**Почему 4 байта padding в конце?**

Структура должна быть кратна выравниванию (8 байт):
- Фактический размер: 16 + 8 + 4 = 28 байт
- До 32 байт: 4 байта padding
- 32 байта / 8 = 4 (кратное выравниванию)

**Зачем?** Если создать массив `[]Person`, каждый элемент будет выровнен по 8 байт.

### Оптимизация памяти

**1. Переупорядочь поля структур:**

```go
// ❌ 48 байт
type Request struct {
    Method  string  // 24
    Active  bool    // 1 + 7 padding
    ID      int64   // 8
    Path    string  // 24
    Code    int32   // 4 + 4 padding
}

// ✅ 40 байт
type Request struct {
    Method  string  // 24
    Path    string  // 24
    ID      int64   // 8
    Code    int32   // 4
    Active  bool    // 1 + 3 padding
}
```

**2. Используй bit fields для флагов:**

```go
// ❌ 3 байта
type Flags struct {
    A bool
    B bool
    C bool
}

// ✅ 1 байт
type Flags byte

const (
    FlagA Flags = 1 << iota
    FlagB
    FlagC
)
```

Про `iota` можно прочитать в главе про [константы](./const.md).

**3. Избегай указателей в маленьких структурах:**

```go
// ❌ Указатель + overhead
type Node struct {
    next *Node  // 8 байт
    val  int    // 8 байт
}

// ✅ Значение
type Node struct {
    nextID int  // 8 байт
    val    int  // 8 байт
}
```

## 🛠 Директивы компилятора

Директивы — это специальные комментарии `//go:...`, которые управляют поведением компилятора.

### `//go:generate`

Запускает команду при `go generate`:

```go
//go:generate stringer -type=Pill
package main

type Pill int
```

### `//go:build`

Условия включения файла в сборку:

```go
//go:build linux && amd64

package platform
```

### `//go:noinline`

Запрет встраивания:

```go
//go:noinline
func benchmark() {
    // не будет встроена
}
```

### `//go:nosplit`

Отключение проверки переполнения стека (только для runtime):

```go
//go:nosplit
func throw(s string) {
    // критический код
}
```

### `//go:linkname`

Переименование символов при линковке:

```go
//go:linkname myAlias runtime.startTheWorld
func myAlias()
```

### `//go:embed`

Встраивание файлов в бинарник:

```go
import "embed"

//go:embed config.json
var config string
```

Полный список директив см. в документации компилятора.

## ⚠️ Типичные проблемы и решения

### Медленная сборка

**Проблема:** `go build` занимает много времени.

**Решения:**

1. Проверить кэш:
   ```bash
   go clean -cache  # очистить кэш
   go build -x      # посмотреть, что компилируется
   ```

2. Уменьшить количество зависимостей:
   ```bash
   go mod tidy
   ```

3. Использовать кэш правильно:
   ```bash
   go env GOCACHE  # убедиться, что кэш включён
   ```

### Большой бинарник

**Проблема:** исполняемый файл слишком большой.

**Решения:**

1. Удалить отладочную информацию:
   ```bash
   go build -ldflags="-s -w"
   ```

2. Проверить зависимости:
   ```bash
   go mod why github.com/large/package
   ```

3. Использовать сжатие:
   ```bash
   upx --best binary
   ```

### Утечки памяти

**Проблема:** программа потребляет много памяти.

**Решения:**

1. Проверить escape analysis:
   ```bash
   go build -gcflags="-m"
   ```

2. Профилировать heap:
   ```bash
   go tool pprof http://localhost:6060/debug/pprof/heap
   ```

3. Избегать глобальных переменных и замыканий с захватом.

## 🎯 Золотые правила

1. **Доверяй компилятору** — он умнее тебя в оптимизациях. Пиши чистый код, а не «умные» оптимизации.
2. **Профилируй перед оптимизацией** — не оптимизируй то, что не измерил.
3. **Понимай escape analysis** — это поможет писать более эффективный код.
4. **Используй inlining с умом** — маленькие функции встраиваются автоматически.
5. **Следи за размером бинарника** — удаляй отладочную информацию в продакшене.
6. **Кэш — твой друг** — не очищай его без необходимости.
7. **Используй ssa.html для глубокой отладки** — когда производительность критична.
8. **Избегай проверок границ в горячих циклах** — используй `range` или явные подсказки.
9. **Читай ssa.html как книгу** — каждый этап рассказывает историю оптимизации твоего кода.
10. **Группируй поля структур по размеру** — от больших к меньшим для уменьшения padding.
11. **Минимизируй аллокации в куче** — GC работает быстрее, когда меньше мусора.
12. **Настраивай GC под свою нагрузку** — GOGC и GOMEMLIMIT могут значительно улучшить производительность.
