# 📦 Пакеты, модули и workspace в Go

Представь, что ты строишь дом. У тебя есть кирпичи, балки, окна, двери — отдельные элементы. Но сам по себе кирпич ничего не значит. Чтобы построить дом, нужно собрать кирпичи в стены, стены в комнаты, а комнаты в здание.

В Go всё устроено похоже:

- **Пакет** — это комната (группа связанных файлов)
- **Модуль** — это здание (набор пакетов с версионированием)
- **Workspace** — это комплекс зданий (несколько модулей вместе)

Давай разберёмся, как всё это работает и зачем нужно.

---

## 🧱 Пакеты (Packages)

**Пакет** — это базовая единица организации кода в Go. Каждый `.go`-файл принадлежит какому-либо пакету, и пакет состоит из одного или нескольких файлов.

### Структура пакета

```
myproject/
├── main.go          // package main
├── utils/
│   ├── helpers.go   // package utils
│   └── math.go      // package utils
└── config/
    └── settings.go  // package config
```

Все файлы в одной папке должны принадлежать одному пакету:

```go
// utils/helpers.go
package utils

func Help() {
    // ...
}

// utils/math.go
package utils  // то же имя пакета

func Calculate() {
    // ...
}
```

Файлы в пакете «видят» всё, что объявлено в других файлах этого же пакета — как экспортируемое, так и нет.

### Именование пакетов

Правила простые, но важные:

```go
package utils      // ✅ хорошо: коротко, понятно, в нижнем регистре
package MyUtils    // ❌ плохо: заглавные буквы
package my_utils   // ❌ плохо: подчёркивания
package util       // ⚠️ неоднозначно: utils лучше

package http       // ✅ стандартная библиотека
package myhttp     // ✅ если очень нужно
```

**Главное правило:** имя пакета должно читаться как префикс для его содержимого:

```go
import "net/http"

http.Client    // ✅ читается естественно
http.Request   // ✅

import "utils"

utils.Buffer   // ✅ понятно
utils.Helper   // ✅
```

Избегай бессмысленных имён вроде `common`, `shared`, `base` — они не говорят о назначении пакета.

### Экспорт через капитализацию

В Go нет ключевых слов `public`, `private`, `protected`. Вместо этого используется гениально простой механизм: **имя с заглавной буквы экспортируется, со строчной — остаётся внутри пакета**.

```go
package config

// Version — экспортируется (заглавная буква)
var Version = "1.0.0"

// version — не экспортируется (строчная буква)
var version = "1.0.0-dev"

// Load — экспортируемая функция
func Load() *Config {
    return &Config{}
}

// load — внутренняя функция
func load() *Config {
    return &Config{}
}
```

```go
package main

import "myproject/config"

func main() {
    _ = config.Version  // ✅ работает
    _ = config.Load()   // ✅ работает
    
    _ = config.version  // ❌ ошибка: cannot refer to unexported name
    _ = config.load()   // ❌ ошибка
}
```

Это правило распространяется на всё: переменные, константы, функции, типы, интерфейсы и даже поля структур:

```go
type User struct {
    Name string  // ✅ экспортируется
    age  int     // ❌ не экспортируется
}
```

### Пакет `main` и точка входа

Пакет с именем `main` — особенный. Это исполняемый файл.

```go
package main

import "fmt"

func main() {
    fmt.Println("Hello, World!")
}
```

**Важно:**
- Только пакет `main` компилируется в исполняемый файл
- Пакет `main` должен содержать функцию `main()`
- Пакет `main` нельзя импортировать другими пакетами

Библиотеки используют другие имена:

```go
package http      // библиотека, можно импортировать
package json      // библиотека, можно импортировать
package mylib     // библиотека, можно импортировать
```

### Функция `init()`

Каждый пакет может содержать одну или несколько функций `init()`. Они вызываются **автоматически** при запуске программы, до `main()`.

```go
package config

import "os"

var Port string

func init() {
    Port = os.Getenv("PORT")
}
```

**Порядок вызова `init()`:**

```
┌─────────────────────────────────────────────────────────┐
│  1. Инициализация импортируемых пакетов                 │
│     (в порядке импорта, рекурсивно)                     │
│     ↓                                                   │
│  2. Вызов init() в текущем пакете                       │
│     (в порядке объявления в файле)                      │
│     ↓                                                   │
│  3. Вызов main()                                        │
└─────────────────────────────────────────────────────────┘
```

Пример:

```go
// a/a.go
package a

import "fmt"

func init() {
    fmt.Println("init A")
}

// b/b.go
package b

import (
    "fmt"
    _ "myproject/a"
)

func init() {
    fmt.Println("init B")
}

// main.go
package main

import (
    "fmt"
    _ "myproject/b"
)

func init() {
    fmt.Println("init main")
}

func main() {
    fmt.Println("main")
}

// Вывод:
// init A
// init B
// init main
// main
```

**⚠️ Предупреждение:** не злоупотребляй `init()`. Скрытая инициализация усложняет понимание кода:

```go
// ❌ Плохо: скрытая логика
package config

var Port string

func init() {
    Port = os.Getenv("PORT")  // откуда взялся Port?
}

// ✅ Лучше: явный конструктор
package config

func Load() *Config {
    return &Config{
        Port: os.Getenv("PORT"),
    }
}
```

### Пакет `internal` — ограничение видимости

Иногда нужно сделать пакет доступным только внутри своего модуля. Для этого существует папка `internal`:

```
myproject/
├── internal/
│   └── db/          // доступен только внутри myproject/...
├── public/
│   └── api/         // доступен всем
└── main.go
```

```go
// main.go
import "myproject/internal/db"  // ✅ работает

// В другом проекте:
import "myproject/internal/db"  // ❌ ошибка: use of internal package not allowed
```

`internal` может быть вложен:

```
myproject/
├── internal/
│   └── db/
│       └── internal/
│           └── driver/  // доступен только внутри internal/db/...
└── main.go
```

Это мощный инструмент для скрытия внутренней архитектуры библиотеки.

## 📥 Импорт пакетов: все нюансы

Импорт в Go кажется простым, но есть несколько важных нюансов, которые нужно знать.

### Базовый (Квалифицированный) импорт

```go
import "fmt"

fmt.Println("Hello")  // квалифицированный импорт
```

Это стандартный способ: ты обращаешься к экспортируемым именам через имя пакета.

### Именованный импорт (алиас)

```go
import m "math"

m.Sin(0.5)  // используем алиас вместо имени пакета
```

**Когда использовать алиасы:**

1. **Конфликт имён пакетов:**

```go
import (
    "github.com/user/project/internal/models"
    modelsV2 "github.com/user/project/v2/models"  // алиас обязателен
)

var m1 models.User
var m2 modelsV2.User
```

2. **Сокращение длинных путей:**

```go
import (
    k8s "k8s.io/api/core/v1"
    metav1 "k8s.io/apimachinery/pkg/apis/meta/v1"
)

pod := &k8s.Pod{
    ObjectMeta: metav1.ObjectMeta{
        Name: "my-pod",
    },
}
```

3. **Улучшение читаемости:**

```go
import (
    "testing"
    
    "github.com/stretchr/testify/assert"
    "github.com/stretchr/testify/require"
)

func TestSomething(t *testing.T) {
    assert.Equal(t, 1, 1)
    require.NoError(t, err)
}
```

**⚠️ Не злоупотребляй алиасами.** Если пакет требует алиаса для понятности, возможно, стоит пересмотреть структуру кода.

### Пустой импорт (`_`)

```go
import _ "github.com/lib/pq"
```

Пустой импорт загружает пакет, но **не предоставляет доступа к его именам**. Это нужно для **побочных эффектов** — обычно для простого запуска `init()`.

**Как это работает:**

```go
// driver/postgres/postgres.go
package postgres

import "database/sql"

func init() {
    sql.Register("postgres", &Driver{})  // регистрация в init()
}

// main.go
package main

import (
    "database/sql"
    _ "github.com/lib/pq"  // импорт только для регистрации
)

func main() {
    db, err := sql.Open("postgres", "conn-string")  // драйвер уже зарегистрирован
    // ...
}
```

**Типичные сценарии использования `_`:**

1. **Драйверы баз данных:**

```go
import _ "github.com/go-sql-driver/mysql"
import _ "github.com/lib/pq"
import _ "github.com/mattn/go-sqlite3"
```

2. **pprof для профилирования:**

```go
import _ "net/http/pprof"

func main() {
    // теперь на /debug/pprof доступны профили
    http.ListenAndServe("localhost:6060", nil)
}
```

3. **Инициализация без использования:**

```go
// Пакет регистрирует обработчики в init()
import _ "myproject/handlers"
```

**Важно:** без `_` компилятор выдаст ошибку `imported and not used`.

---

### Точечный импорт (`.`)

```go
import . "fmt"

Println("Hello")  // без префикса пакета
```

Точечный импорт «растворяет» имена пакета в текущей области видимости.

**⚠️ Не используй точечный импорт в продакшене.** Это:
- Ухудшает читаемость (непонятно, откуда взялось имя)
- Может вызвать конфликты имён
- Считается плохим стилем

**Когда допустимо:**

1. **В тестах:**

```go
package main_test

import (
    "testing"
    . "github.com/onsi/ginkgo/v2"
    . "github.com/onsi/gomega"
)

var _ = Describe("Test", func() {
    It("should work", func() {
        Expect(1).To(Equal(1))  // имена из пакета без префикса
    })
})
```

2. **В примерах кода:**

```go
// example_test.go
package main_test

import (
    "fmt"
    . "math"  // для удобства в примерах
)

func ExampleSin() {
    fmt.Println(Sin(0.5))  // без math.
    // Output: 0.479425538604203
}
```

### Группировка импортов

Рекомендуется группировать импорты в три блока:

```go
import (
    // Стандартная библиотека
    "context"
    "fmt"
    "net/http"
    "time"

    // Сторонние зависимости
    "github.com/google/uuid"
    "github.com/stretchr/testify/assert"
    "golang.org/x/sync/errgroup"

    // Внутренние пакеты проекта
    "myproject/internal/config"
    "myproject/pkg/logger"
    "myproject/service/user"
)
```

**Почему это важно:**
- Улучшает читаемость
- Легко найти нужный импорт
- `go fmt` и `goimports` автоматически форматируют импорты

---

### Факторизованный стиль

Используй группировку в одной скобке:

```go
// ✅ Хорошо
import (
    "fmt"
    "math"
)

// ❌ Плохо (устаревший стиль)
import "fmt"
import "math"
```

**Преимущества факторизованного стиля:**
- Удобно добавлять и удалять пакеты
- Можно группировать по логике
- Это стандарт де-факто

---

### Удаление неиспользуемых импортов

Компилятор Go выдаёт ошибку, если импорт не используется:

```go
import "fmt"  // ❌ ошибка: imported and not used

func main() {}
```

**Решение:**
- Удали неиспользуемый импорт
- Или используй `_` для побочных эффектов

**Автоматическая очистка:**

```bash
go mod tidy  # удалит неиспользуемые зависимости из go.mod
goimports -w .  # удалит неиспользуемые импорты из файлов
```

---

### Неиспользуемый импорт в разработке

При разработке иногда нужно временно отключить использование пакета, но оставить импорт:

```go
import "fmt"

func main() {
    _ = fmt.Println  // «использование» для компилятора
    // основной код закомментирован
}
```

Это костыль, но помогает при отладке.

## 🔁 Циклический импорт и как с ним бороться

Go **не разрешает циклические импорты**:

```go
// a/a.go
package a

import "myproject/b"  // ❌

// b/b.go
package b

import "myproject/a"  // ❌ цикл!
```

**Ошибка:** `import cycle not allowed`.

**Почему это важно:** циклические зависимости усложняют поддержку и тестирование.

### Решение: инверсия зависимостей

Вынеси общий интерфейс в отдельный пакет:

```go
// model/user.go
package model

type User struct {
    ID   int
    Name string
}

// storage/db.go
package storage

import "myproject/model"

type DB interface {
    GetUser(id int) (*model.User, error)
}

// service/service.go
package service

import (
    "myproject/model"
    "myproject/storage"
)

type Service struct {
    db storage.DB
}

func (s *Service) Fetch(id int) (*model.User, error) {
    return s.db.GetUser(id)
}
```

Теперь зависимости направлены в одну сторону:

```
model ← storage ← service
```

### Области видимости в контексте пакетов

Пакеты тесно связаны с областями видимости. В Go используется **лексическая область видимости** — она определяется на этапе компиляции по тому, где в коде объявлен идентификатор.

**Иерархия областей:**

```
┌─────────────────────────────────────┐
│           Universe                  │  ← built-in: len, int, nil
│  ┌───────────────────────────────┐  │
│  │         Package               │  │  ← переменные пакета
│  │  ┌─────────────────────────┐  │  │
│  │  │       Function          │  │  │  ← локальные переменные
│  │  │  ┌───────────────────┐  │  │  │
│  │  │  │       Block       │  │  │  │  ← if, for, { }
│  │  │  └───────────────────┘  │  │  │
│  │  └─────────────────────────┘  │  │
└─────────────────────────────────────┘
```

**Universe** — самая внешняя область. В ней живут все встроенные сущности: `int`, `string`, `len`, `make`, `nil`, `true`, `false`.

**Package scope** — всё, что объявлено вне функций, видно во всём пакете (во всех файлах пакета).

**В Go нет области видимости файла** — файл не создаёт отдельную область. Всё объявленное на уровне пакета в одном файле видно во всех файлах этого пакета.

```go
// config/version.go
package config

var Version = "1.0.0"

// config/settings.go
package config

func GetVersion() string {
    return Version  // ✅ работает, хотя в другом файле
}
```

## 📦 Модули (Modules)

**Модуль** — это набор пакетов с версионированием. Модуль описывается в файле `go.mod` и является минимальной единицей управления зависимостями в Go.

### История: что было до модулей

До Go 1.11 весь код должен был лежать в `$GOPATH/src`:

```
$GOPATH/
└── src/
    ├── github.com/
    │   └── user/
    │       └── project/
    └── myproject/
```

**Проблемы:**
- Все проекты делили одни зависимости
- Нельзя было иметь разные версии одной библиотеки
- Требовалось интернет-соединение для сборки
- Сложно работать без доступа к репозиториям

Модули решили эти проблемы.

### Структура модуля

```
myproject/
├── go.mod         # Описание модуля и зависимостей
├── go.sum         # Контрольные суммы зависимостей
├── main.go
├── internal/      # Внутренние пакеты
└── pkg/           # Публичные пакеты
```

### Файл `go.mod`

```gomod
module github.com/myorg/myproject

go 1.22

require (
    github.com/google/uuid v1.3.0
    github.com/stretchr/testify v1.8.0 // indirect
)

replace github.com/old/lib => ../new/lib

exclude github.com/bad/lib v1.1.0

retract v1.0.0
```

**Директивы:**

| Директива | Описание |
|-----------|----------|
| `module` | Имя модуля (путь импорта) |
| `go` | Минимальная версия Go |
| `require` | Зависимости модуля |
| `replace` | Замена зависимости на другую версию или путь |
| `exclude` | Исключить конкретную версию |
| `retract` | Отозвать версию (для своих модулей) |
| `toolchain` | Принудительная версия тулчейна |
| `godebug` | Настройки отладки |

**`require` с `// indirect`:**

```gomod
require (
    github.com/google/uuid v1.3.0           // прямая зависимость
    github.com/stretchr/testify v1.8.0 // indirect  // транзитивная
)
```

`// indirect` означает, что эта зависимость используется не напрямую в коде, а через другую зависимость.

### Файл `go.sum`

`go.sum` содержит хэши модулей для проверки целостности:

```
github.com/google/uuid v1.3.0 h1:t6JcX0F+Vn90vG8bVn1fGf9G8rX8H8L8V8H8H8H8H8=
github.com/google/uuid v1.3.0/go.mod h1:TIyPZe4MgqvfeYDBFedMoGGpEw/LqOeaOT+nhxU+yHo=
```

**Не редактируй `go.sum` вручную!** Он обновляется автоматически при:
- `go get`
- `go mod tidy`
- `go build` (при загрузке новых зависимостей)

### Управление зависимостями

**Инициализация модуля:**

```bash
go mod init github.com/myorg/myproject
```

**Синхронизация зависимостей:**

```bash
go mod tidy  # добавляет недостающие, удаляет лишние
```

**Загрузка зависимостей:**

```bash
go mod download  # загружает в кэш
```

**Почему модуль не загружается:**

```bash
go mod why github.com/pkg/errors  # покажет, кто тянет зависимость
```

**Проверка целостности:**

```bash
go mod verify  # сверяет хэши с go.sum
```

### Семантическое версионирование

Go использует SemVer:

```
v1.2.3
│ │ │
│ │ └─ Patch (исправление ошибок)
│ └─── Minor (новая функциональность)
└───── Major (обратно несовместимые изменения)
```

**Важно:** начиная с v2, мажорная версия должна быть в пути модуля:

```gomod
// ❌ Неправильно для v2+
module github.com/myorg/mylib

// ✅ Правильно для v2+
module github.com/myorg/mylib/v2
```

```go
import "github.com/myorg/mylib/v2"
```

### Приватные модули: `GOPRIVATE`

По умолчанию Go использует публичные прокси (`proxy.golang.org`). Для приватных репозиториев нужно настроить `GOPRIVATE`:

```bash
go env -w GOPRIVATE=github.com/myorg/*,gitlab.company.com/*
```

Теперь Go будет:
- Не использовать прокси для этих путей
- Не проверять контрольные суммы в `sum.golang.org`
- Обращаться напрямую к репозиторию по SSH/HTTPS

### `replace` — замена зависимостей

```gomod
replace github.com/example/lib => ../local/lib
replace github.com/example/lib v1.2.3 => github.com/fork/lib v1.2.4-fixed
```

**Когда использовать:**
- Локальная разработка зависимостей
- Использование форков
- Временные исправления

**⚠️ Важно:** `replace` не распространяется на другие модули — это локальная настройка.

## 🏢 Workspace (`go.work`)

**Workspace** — это механизм для работы с несколькими модулями одновременно. Появился в Go 1.18.

### Зачем нужен workspace

Представь, что у тебя есть:
- Основное приложение `myapp`
- Библиотека `mylib`, которую ты разрабатываешь параллельно

Без workspace тебе пришлось бы постоянно публиковать новые версии `mylib` и обновлять `go.mod` в `myapp`. С workspace ты можешь работать с обоими модулями одновременно.

### Структура workspace

```
workspace/
├── go.work        # Конфигурация workspace
├── myapp/
│   └── go.mod
└── mylib/
    └── go.mod
```

```gowork
go 1.22

use (
    ./myapp
    ./mylib
)
```

Теперь `myapp` будет использовать локальную версию `mylib`, даже если в `go.mod` указана другая версия.

### Создание workspace

```bash
# Инициализация
go work init ./myapp ./mylib

# Добавление модуля
go work use ./another-lib

# Синхронизация
go work sync
```

### `go.work` vs `replace`

| Критерий | `replace` в `go.mod` | `go.work` |
|----------|---------------------|-----------|
| Где используется | В каждом `go.mod` | Один `go.work` для всех |
| Область действия | Только текущий модуль | Все модули в workspace |
| Коммитится в репо | Да (если нужно команде) | Обычно нет (локально) |
| Для чего | Форки, постоянные замены | Разработка, монорепы |

**Можно использовать вместе:**

```gowork
# go.work
use (
    ./myapp
    ./mylib
)
```

```gomod
# myapp/go.mod
require github.com/ext/foo v1.2.0

replace github.com/ext/foo => ../forks/foo  // будет переопределено go.work
```

`use` из `go.work` имеет приоритет над `replace`.

### `go.work.sum`

При использовании workspace создаётся файл `go.work.sum` — аналог `go.sum` для всего workspace.

## 📂 Vendor

Каталог `vendor/` содержит копии всех зависимостей:

```bash
go mod vendor  # создаёт vendor/
go build -mod=vendor  # использует vendor/
```

**Когда использовать:**
- CI/CD без доступа к интернету
- Полная гарантия воспроизводимости
- Приватные зависимости без настройки `GOPRIVATE`

**Когда не использовать:**
- Обычная разработка (проще через прокси)
- Если размер репозитория важен (`vendor` может весить 100MB+)

## 🔄 Как Go выбирает версии: Minimal Version Selection

Go использует алгоритм **MVS (Minimal Version Selection)**:

```
┌─────────────────────────────────────────────────────────┐
│  Модуль A требует: lib v1.2.0                           │
│  Модуль B требует: lib v1.1.0                           │
│                                                         │
│  Go выберет: lib v1.2.0 (наибольшую совместимую)        │
└─────────────────────────────────────────────────────────┘
```

В отличие от npm или Maven, Go не ищет «последнюю совместимую» версию — он выбирает **минимальную версию, удовлетворяющую всем требованиям**. Это делает сборки воспроизводимыми.

## 🧬 Внутреннее устройство: как компилятор видит пакеты

Когда компилятор обрабатывает код, каждый пакет представлен структурой `types.Package`:

```go
type Package struct {
    path      string        // путь импорта: "fmt", "github.com/user/repo/pkg"
    name      string        // имя пакета: "fmt", "repo"
    scope     *Scope        // область видимости пакета
    imports   []*Package    // импортируемые пакеты
    complete  bool          // полностью ли загружен
    cgo       bool          // используется ли cgo
    goVersion string        // минимальная версия Go
}
```

**`scope`** содержит все объявления верхнего уровня: функции, типы, переменные, константы.

**`imports`** — список всех импортируемых пакетов (прямых и косвенных).

Когда ты пишешь `import "fmt"`, компилятор:
1. Находит пакет `fmt` по пути
2. Загружает его `scope`
3. Добавляет в `imports` текущего пакета
4. Разрешает имена через `fmt.Println`

### Scope: область видимости в компиляторе

**`types.Scope`** — это структура, представляющая область видимости в компиляторе Go.

```go
type Scope struct {
    parent   *Scope          // родительская область
    children []*Scope        // дочерние области
    number   int             // уникальный номер
    elems    map[string]Object  // имена → объекты в этой области
    pos, end token.Pos       // начало и конец области в коде
    comment  string          // комментарий (для отладки)
    isFunc   bool            // это область функции?
}
```

**Как это работает:**

```
┌─────────────────────────────────────────────────────────────┐
│  Universe Scope (корневая)                                  │
│  └── Package Scope (mypackage)                              │
│      ├── Function Scope (main)                              │
│      │   └── Block Scope (if/for)                           │
│      ├── Function Scope (helper)                            │
│      └── Type Scope (MyStruct)                              │
└─────────────────────────────────────────────────────────────┘
```

**Поля Scope:**

| Поле | Тип | Описание |
|------|-----|----------|
| `parent` | `*Scope` | Ссылка на внешнюю область. Через неё компилятор поднимается вверх при поиске имени |
| `children` | `[]*Scope` | Список вложенных областей |
| `number` | `int` | Уникальный номер в пределах родительской области |
| `elems` | `map[string]Object` | Карта всех имён, объявленных в этой области |
| `pos`, `end` | `token.Pos` | Позиция в исходном коде, где область начинается и заканчивается |
| `isFunc` | `bool` | true, если это область функции |

**Поиск имени:**

Когда компилятор встречает имя в коде, он:

```
┌──────────────────────────────────────────┐
│  1. Ищет в текущей области (Scope)       │
│     ↓ не найдено                         │
│  2. Идёт в parent (родительскую)         │
│     ↓ не найдено                         │
│  3. Повторяет, пока не достигнет верха   │
│     ↓ не найдено                         │
│  4. Проверяет Universe                   │
│     ↓ не найдено                         │
│  5. Ошибка: undefined                    │
└──────────────────────────────────────────┘
```

Этот процесс называется **resolution** (разрешение имён).

### Object: именованные сущности

Каждое имя в Scope представлено интерфейсом `types.Object`:

```go
type Object interface {
    Parent() *Scope      // Область видимости
    Pos() token.Pos      // Позиция в исходнике
    Pkg() *Package       // Пакет
    Name() string        // Имя
    Type() Type          // Тип
    Exported() bool      // Экспортируется ли
    Id() string          // Имя (экспортированное) или квалифицированное имя
}
```

**Метод `Id()`:**

- Для **экспортируемых** объектов: возвращает имя (`"MyType"`)
- Для **неэкспортируемых** объектов: возвращает квалифицированное имя (`"pkg.MyType"`)

```go
// В пакете "mypkg"
type MyType struct{}      // Exported() = true,  Id() = "MyType"
type myType struct{}      // Exported() = false, Id() = "mypkg.myType"
```

**Зачем нужен `Id()`:**

Когда ты работаешь с несколькими пакетами, `Id()` позволяет уникально идентифицировать неэкспортируемые объекты:

```go
// В main.go
import "mypkg"

var x mypkg.MyType   // x.Type().Obj().Id() = "MyType"
var y mypkg.myType   // y.Type().Obj().Id() = "mypkg.myType"
```

**Конкретные реализации:**

| Тип | Описание |
|-----|----------|
| `*types.Var` | Переменная (локальная, параметр, поле) |
| `*types.Func` | Функция или метод |
| `*types.TypeName` | Имя типа (struct, interface) |
| `*types.Const` | Константа |
| `*types.Label` | Метка для goto/break/continue |

**Пример: как компилятор видит переменную:**

```go
func main() {
    x := 42
}
```

В компиляторе это:

```
Scope (main function)
└── elems["x"] → *types.Var{
        name: "x",
        type: int,
        pos: main.go:2:5,
        kind: LocalVar
    }
```

**Зачем это знать?** Если ты пишешь инструменты для Go (линтеры, анализаторы), ты будешь работать с этими структурами. Для обычного кода достаточно понимать: компилятор видит переменные не как «ячейки памяти», а как объекты с метаданными.

## 🎯 Золотые правила

1. **Один пакет = одна папка**. Не создавай разные пакеты в одной папке.
2. **Именуй пакеты понятно**. `utils` — плохо, `httpclient` — хорошо.
3. **Избегай `init()`**. Явные конструкторы лучше скрытой инициализации.
4. **Используй `internal`** для скрытия внутренней архитектуры.
5. **Не коммить `go.work`** — это локальный файл для разработки.
6. **`go mod tidy` перед коммитом** — убирай неиспользуемые зависимости.
7. **Избегай циклических импортов** — выноси интерфейсы в отдельные пакеты.
8. **Не используй точечный импорт (`.`)** в продакшене — только в тестах.
9. **Группируй импорты**: стандартная библиотека → сторонние → внутренние.
10. **Используй `_` только для побочных эффектов** — драйверы, pprof, регистрация.

Пакеты, модули и workspace — это фундамент организации кода в Go. Понимание того, как они работают вместе, поможет тебе писать чистые, поддерживаемые проекты любого размера.
