    📚 Документирование кода в Go

    Go делает ставку на простоту, читаемость и тесную связь документации с кодом.

    Комментарии не просто украшают исходники, они — часть интерфейса и ключ к поддержке, читаемости и генерации автодокументации через godoc и pkg.go.dev.

    GoDoc — это интерфейс

    В Go принято думать о документации как о неявной части публичного API.

    Каждый экспортируемый объект должен «разговаривать» с потребителем через go doc.

    📌 Виды комментариев

    Go поддерживает несколько видов комментариев:

    • // single-line — используются чаще всего и обрабатываются godoc,
    • /* multi-line */ — допускаются, но не рекомендуются в публичной документации,
    • //go: — директивы компилятора и инструментов (см. ниже).

    Подробнее про //go: и директивы компилятора см. в главе 👉 Директивы.

    // Println formats using the default formats for its operands and writes to standard output.
func Println(a ...any) (n int, err error) {}

/* This is a block comment.
   It will not be shown in GoDoc. */

    Предупреждение

    Не все комментарии попадают в godoc.

    Только //-комментарии, расположенные непосредственно над экспортируемыми объявлениями, включаются в итоговую документацию.

    🧠 Философия GoDoc

    1. Комментарии отображаются в документации, если они:

      • находятся непосредственно над экспортируемыми сущностями (типами, функциями, переменными и т.д.),
      • начинаются с имени документируемого объекта (например: // Reader ...).

    2. Вся документация доступна онлайн через pkg.go.dev или локально с помощью godoc.

    3. Стиль — простой, «как если бы документации не существовало», без особых тегов и аннотаций.

      Рекомендация

      Используйте полный первый заголовочный комментарий в виде предложения, начинающегося с имени сущности:

      • // Reader provides ...
      • // ParseConfig returns ...

    📦 Документирование

    Что такое док-комментарий

    Док-комментарий — это комментарий, который появляются непосредственно перед объявлениями пакета, константы, функции, типа и переменной верхнего уровня без промежуточных новых строк.

    Каждое экспортируемое (с большой буквы) имя должно иметь док-комментарий.

    🧱 Пакеты

    Для того, чтобы задокументировать пакет, надо прописать комментарий хотя бы в одном файле пакета.

    // Package sort provides primitives for sorting slices and user-defined collections.
package sort

    Рекомендация

    Рекомендуется использовать doc.go с общим описанием пакета, примерами использования, ограничениями и предупреждениями.

    🧑‍💻 Команды

    Комментарии к package main обрабатываются godoc, как и у обычных пакетов. Они используются для документирования поведения команды, которую реализует программа.

    Такие комментарии отображаются в заголовке документации пакета на pkg.go.dev, и особенно полезны, если ваша команда предоставляет утилиту командной строки или инструмент.

    // Gofmt formats Go programs.
// It uses tabs for indentation and spaces for alignment.
package main

    Совет

    Даже для main.go стоит писать описание поведения команды — это помогает пользователям и поддерживающим в будущем.

    🧩 Типы и структуры

    Комментарии к структурам и типам особенно важны, потому что они задают контракт и ожидания от использования объекта.

    Примечание

    Хороший комментарий должен чётко объяснять, для чего предназначен тип, какие методы он реализует (если это не очевидно из кода), и какие гарантии он даёт.

    Особое внимание стоит уделять:

    • Назначению и области применения,
    • Поведению нулевого значения,
    • Потокобезопасности (если необходимо),
    • Ограничения, побочные эффекты или нестандартные способы использования.

    Плохо:

    type Manager struct {
    db *sql.DB
}

    Хорошо:

    // Manager coordinates user operations backed by a SQL database.
// It is safe for concurrent use by multiple goroutines.
type Manager struct {
    db *sql.DB
}

    🔁 Методы и интерфейсы

    Комментарии к методам и интерфейсам помогают другим разработчикам понять назначение методов, ожидаемое поведение, контракты интерфейсов и возможные побочные эффекты.

    🧩 Интерфейсы

    Примечание

    Интерфейс задает контракт и его документация должны быть крайне исчерпывающей.

    // Reader wraps the basic Read method.
type Reader interface {
    // Read reads up to len(p) bytes into p.
    // It returns the number of bytes read and any error encountered.
    Read(p []byte) (n int, err error)

    ...
}

    Если интерфейс небольшой, то допустим вариант без описания каждого метода.

    // Closer is implemented by anything that can be closed.
type Closer interface {
    Close() error
}

    🔨 Методы

    Предупреждение

    Избегайте комментариев-повторов.

    Комментарий вида // Read reads ... избыточен, если поведение полностью понятно из сигнатуры.

    Однако если реализация добавляет дополнительную логику (например, кэширование, буферизацию, retry-механику), это следует задокументировать.

    Примечание

    Комментарий метода должен описывать строго поведение, а не реализацию."

    // Read reads data from the socket and buffers it internally before returning.
// This adds latency but improves throughput.
func (r *BufferedReader) Read(p []byte) (int, error) {
    ...
}

    🧮 Константы и переменные

    Примечание

    Для глобальных переменных и констант также рекомендуется использовать док-комментарии.

    // Version is the Unicode version used in this package.
const Version = "14.0.0"

const (
    // EOF is returned at end of input.
    EOF = -(iota + 1)
    // Ident represents an identifier token.
    Ident
)

    Рекомендация

    При использовании const (...) снабжайте каждую экспортируемую константу отдельным комментарием, даже если они идут подряд.

    🧪 Example* функции

    Что такое Example* функции

    В Go есть такой механизм, что если в *_test.go файлах есть функции Example*, то есть функции будут использоваться внутри godoc для демонстрации.

    Также go test запускает Example*-функции и проверяет, что вывод совпадает с // Output:. Это помогает гарантировать актуальность документации.

    Формат имени Пояснение
    Example Общий пример для пакета
    ExampleFoo Пример для функции Foo
    ExampleBar_Baz Пример для метода Bar.Baz()
    Example_type Пример использования типа (редко)

    Предположим есть следующий код.

    package mypkg

type user struct {
    name string
    Age int
}

func NewUser() *user {
    return &user{}
}

    Тогда можно сделать ExampleNewUser.

    package mypkg_test

import (
    "fmt"
    "testgo/mypkg"
)

func ExampleNewUser() {
    user := mypkg.NewUser()
    fmt.Println(user)
    // Output: &{ 0}
}

    Совет

    Используйте Example-функции как живую документацию.

    Это особенно полезно при работе с SDK, CLI-библиотеками и утилитами.

    🧾 Форматирование

    Комментарии в Go — это не просто текст, они форматируются и структурируются gofmt-ом и становятся частью документации на pkg.go.dev.

    Они должны быть читаемы и в исходниках, и в HTML, и в CLI (go doc), поэтому Go использует упрощённый Markdown-подобный стиль, который:

    • легко писать вручную,
    • не требует дополнительных тегов.

    Стоит следовать следующим правилам:

    • параграфы разделяются пустой строкой,
    • заголовки начинаются с #,
    • списки могут быть маркерованными или нумерованными,
    • [Имя] или [пакет.Имя] создает ссылку на экспортируемый объект,
    • именованные ссылки работают также как и в Markdown.

    Рекомендация

    Используйте заголовки (# Notes, # Error Handling) только в комментариях к интерфейсам или сложным типам.

    Избегайте Markdown-избыточности в коротких описаниях.

    🗂️ Специальные теги

    Эти теги распознаются godoc и отображаются в отдельных разделах:

    🏷️ Тег 📄 Обрабатывается godoc / pkg.go.dev? 📌 Описание
    BUG(...) ✅ Указывает на известную ошибку в пакете или методе. Показывается в разделе "Bugs".
    TODO(...) ✅ Напоминание о задаче, которую нужно выполнить. Показывается в разделе "ToDo" (если парсер godoc включён соответствующим образом).
    Deprecated ✅ Помечает API как устаревший. Должен начинаться с Deprecated: с заглавной буквы и двоеточием.
    Example... ✅ Отображается как пример в документации (ExampleFoo, Example_Bar, и т.д.).
    Output ✅ Последняя строка Example-функции, показывающая ожидаемый вывод.
    GOOS / GOARCH ⚠️ Нет Используется в директивах //go:build, но не является тегом документации.
    +build ⚠️ Нет Устаревший build-тег — также не обрабатывается godoc.

    Предупреждение

    Формат Deprecated: обязателен. Комментарий должен начинаться с Deprecated:, иначе pkg.go.dev не отобразит его как устаревший.

    Пример хорошо оформленного комментария для интерфейса

    // Writer is implemented by objects that can write data.
//
// # Usage
//
// This interface is used in combination with [io.Reader] for streaming
// data in and out of components.
//
// Implementations must be safe for concurrent use by multiple goroutines.
//
// # Error Handling
//
// Implementations should return an error if the write fails.
// If partial writes are allowed, they must return the number of bytes written.
//
// # Notes
//
// * If a type wraps another Writer, it should propagate errors.
// * Zero values of implementations should be usable where possible.
//
// BUG(john): Panics if buffer exceeds 2GB.
//
// TODO(alice): Support for WriteContext(ctx, data) in v2.
//
// Deprecated: WriterV1 is deprecated. Use WriterV2 with context support.
type Writer interface {
    // Write writes len(p) bytes from p to the underlying data stream.
    // It returns the number of bytes written and any error encountered.
    Write(p []byte) (n int, err error)
}

    🧰 Внешние инструменты

    Хотя Go ориентирован на минимализм, в крупных проектах применяются внешние генераторы документации.

    Наиболее популярные из них - Swagger/OpenAPI.

    Здесь не будут рассматриваться данные инструменты.

    ✅ Общие рекомендации

    1. Избегайте очевидных и тривиальных фраз.

      Плохо: // Add adds two integers.

      Лучше: // Add returns the sum of two integers. It panics on overflow.

    2. Упоминайте конкретные ошибки, которые может вернуть функция.

      // ParseConfig returns ErrInvalidFormat if the input doesn't match expected structure.

    3. Указывайте можно ли безопасно использовать тип без инициализации.

      // Buffer is a dynamically-sized buffer.
// The zero value is ready to use.

    4. Слишком тривиальным функциям и методам комментарии не нужны.

    5. В описаниях функций избегайте "how", а фокусируйтесь на "what" и "why".
    6. Не бойтесь писать многострочные комментарии. Главное — передать поведение, контракт и ограничения. Это сэкономит десятки часов будущим разработчикам.
